A complete beginner's tutorial — from zero to confident collaborator

What Are Git and GitHub?

If you've ever saved multiple copies of a file — report_v1.docx, report_v2.docx, report_FINAL.docx, report_FINAL_really.docx — you already understand the problem that Git solves.

Git: Version Control for Your Code

Git is a free, open-source version control system that tracks every change you make to your files. Instead of keeping dozens of copies, Git records a complete history so you can:

  • Go back to any earlier version of your project
  • See exactly what changed, when, and who changed it
  • Work on new features without breaking the working code
  • Collaborate with others without overwriting each other's work

GitHub: Git in the Cloud

GitHub is a website (github.com) that hosts your Git repositories online. Think of it as Google Drive for code — but with superpowers:

  • Your code is backed up in the cloud
  • Teammates can access, review, and contribute to your code
  • You can view the full change history in a browser
  • You can open Pull Requests to propose and review changes
Git vs. GitHub: Git runs on your computer and tracks changes locally. GitHub is an online service where you store and share that history. Your IDE talks to both — you rarely need to think about the distinction.

Getting Set Up

Step 1 — Install Git

Git must be installed on your computer. This is the one step that requires downloading a program — your IDE handles everything else.

macOS

Git is bundled with Xcode. If you have Xcode installed, you already have Git. If not, open Terminal (find it in Applications → Utilities) and run:

xcode-select --install

A dialog appears asking to install the Xcode Command Line Tools — click Install. Git is included. You can close Terminal once it finishes.

Windows

Download the installer from git-scm.com/download/win and run it. Accept all defaults on every screen. No further configuration is needed — your IDE will detect Git automatically.

Linux

Open a terminal and run: sudo apt update && sudo apt install git -y (Ubuntu/Debian), then close the terminal.

Step 2 — Create a GitHub Account

  1. Go to github.com and click Sign up
  2. Choose a username (this is public — pick something professional)
  3. Enter your email and create a password
  4. Verify your account via the email GitHub sends you

Step 3 — Connect Your IDE to GitHub

Each IDE can log into GitHub directly using your browser — no passwords or SSH keys needed. Follow the steps for your IDE below. Once connected, your IDE handles all authentication automatically every time you push or pull.

JetBrains Rider & CLion

  1. Open the IDE → Settings (Cmd+, on Mac / Ctrl+Alt+S on Windows)
  2. Go to Version Control → GitHub
  3. Click + → Log In via GitHub
  4. Your browser opens a GitHub authorization page — click Authorize JetBrains
  5. Return to the IDE — your account now appears in the list. Click OK.

Xcode

  1. Open Xcode → Settings (Cmd+,) → Accounts tab
  2. Click + → GitHub → Continue
  3. Enter your GitHub username and a Personal Access Token (PAT)
Creating a Personal Access Token for Xcode: In your browser, go to GitHub → profile photo → Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → Generate new token. Give it a note (e.g. "Xcode"), check the repo scope, and click Generate token. Copy it immediately — GitHub only shows it once. Paste it into the Xcode Accounts dialog.

Configure Your Name and Email (Rider & CLion)

Your name and email are attached to every commit so teammates know who made each change. Set them in the IDE:

  1. Settings → Version Control → Git
  2. Click Configure user name and email…
  3. Enter your name and the email address you used to sign up for GitHub
  4. Click OK

Xcode uses the name and email from your GitHub account automatically once you're signed in.

Core Git Concepts

Before using Git inside an IDE, it helps to understand the vocabulary. Everything below maps directly to what you'll click in Rider, CLion, and Xcode.

The Repository (Repo)

A repository is a folder that Git is tracking. It stores your files and a complete record of every change ever made. Your IDE shows you the whole history visually in its Git panel.

  • Local repo — lives on your computer. Your IDE reads and writes to it directly.
  • Remote repo — lives on GitHub. Your IDE pushes and pulls to keep them in sync.

Commits and Staging

A commit is a saved snapshot of your project — like a save point in a video game. Before committing, you stage the specific changes you want to include. Your IDE shows all changed files as checkboxes; you check the ones you want, write a message, and click Commit.

Use the imperative mood for commit messages: "Add login button" not "Added login button." A good message completes the sentence: "If applied, this commit will… [your message]."

Branches

A branch is an independent line of development. The default branch is called main. The golden rule is: never commit directly to main. Instead, create a branch for each feature or fix, do your work there, and merge it back when ready.

Your IDE shows the current branch name in the toolbar at the top of the window. Click it at any time to create or switch branches.

Branches are instant — Git doesn't copy your files. They let multiple people (or multiple features) develop in parallel without interfering with each other.

Pull Requests (PRs)

A Pull Request is a GitHub feature (not a Git feature) that lets you propose merging your branch into main. It gives teammates a chance to review your code, leave comments, and approve it before it's merged. PRs are created and reviewed in your browser on github.com.

The Basic Git Workflow

Every day of development follows the same loop. The steps below use github.com for repo creation and your IDE for everything else. Detailed IDE-specific instructions are in the sections that follow.

1. Create a new repo on GitHub

  1. On github.com, click the + icon → New repository
  2. Give it a name (e.g. my-project), set it Public or Private
  3. Check Add a README file
  4. Optionally choose a .gitignore template for your language (e.g. C++, Swift)
  5. Click Create repository

2. Clone the repo into your IDE

Open your IDE and use its built-in clone dialog to download the repo to your computer. See the IDE-specific sections below for exact steps. Your IDE connects to GitHub, lists your repos, and handles the download automatically.

3. Create a feature branch

Click the Git Branch button in the Git Branch button (top toolbar) (shows the current branch name) and choose New Branch. Give it a descriptive name like add-login-screen. Never work directly on main.

4. Write your code

Make your changes in the editor as normal. Your IDE continuously tracks which files have changed, showing them highlighted in the Project panel.

5. Stage and commit

Open the IDE's Commit panel, check the files you want to include, write a clear commit message, and click Commit. Repeat steps 4–5 as often as you like — small, focused commits are better than large ones.

6. Push to GitHub

Use Git → Push in your IDE (or the keyboard shortcut). This uploads your commits to GitHub so they're backed up and visible to teammates.

7. Merge Your Branch (or Open a Pull Request)

A Pull Request (PR) is a formal proposal to merge your feature branch into main. On a team it gives colleagues a chance to review and discuss your code before it lands. As a solo developer you can skip the PR entirely and merge directly in your IDE:

  1. Switch back to main in the branch switcher (Git Branch button (top toolbar))
  2. In Rider or CLion: open the Git tool window → right-click your feature branch → Merge '[branch]' into 'main'
  3. Push to GitHub as normal (VCS → Push)

If you do want a Pull Request — for example, once you start collaborating — Rider and CLion have this built in. Open the Git tool window and look for the Pull Requests tab. You can create, review, and merge PRs there without leaving the IDE. In Xcode, you'll need to go to github.com for this step.

8. Update your local main

Switch back to main in your IDE's branch switcher, then use Git → Pull to download the merged changes. You're now ready to start the next feature branch.

Always pull from main before creating a new branch. This ensures your work starts from the latest code and keeps merge conflicts to a minimum.

GitHub in JetBrains Rider 2026.1.2

Rider is a cross-platform .NET and C# IDE. All Git operations are available through the menus, the Git tool window, the Git Branch button (top toolbar), and the Commit panel. You rarely need the terminal for anything.

  2. Select GitHub in the left panel — your repos appear automatically
  3. Click the repo you want, choose a local directory, then click Clone

If a repo isn't yours (e.g. an open-source project), paste its URL directly into the URL field at the top of the Clone dialog.

  2. Click New Branch

Switch to an existing branch:

  2. Under Remote Branches, find the branch you want
  3. Click it → Checkout

  2. You'll see a list of all changed files with checkboxes — check the ones to include
  3. Click any file to see an inline diff of exactly what changed
  4. Write a commit message in the text field at the bottom
  5. Click Commit to save locally, or Commit and Push to save and upload to GitHub in one step
Review the diff for every file before committing. The habit of reading your own changes catches a surprising number of mistakes before anyone else sees them.

GitHub in JetBrains CLion 2026.1.3

CLion is a C and C++ IDE. Its Git UI is identical to Rider — the same menus, the same keyboard shortcuts, the same Git tool window. Everything in the Rider section above applies here too. This section covers CLion-specific details.

Complete Branch Workflow

This is the full sequence from creating a branch to merging it back into main. Follow these steps in order every time. The key rule: you must be on main before you can merge into it.

1. Confirm you are on main

Look at the Git Branch button in the top toolbar. It shows the name of the branch you are currently on. If it says main, continue. If it shows another branch name, click it → find main in the list → click the > arrow → click Checkout.

2. Create a new branch

  1. Click the Git Branch button in the top toolbar
  2. Click New Branch…
  3. Type a descriptive name (e.g. add-login-screen) — no spaces, use hyphens
  4. Click Create

The Git Branch button now shows your new branch name, confirming you are on it.

3. Make your changes

Edit your files in the editor as normal. CLion highlights changed files in the Project panel so you can see what's been modified.

4. Save your files

File → Save All (or ⌘S). Always save before committing.

5. Commit your changes

  1. Click the Git Branch button → Commit… (or press ⌘K)
  2. The Commit dialog opens showing all changed files — make sure the ones you want are checked
  3. Type a short, clear commit message describing what you changed (e.g. Add login screen layout)
  4. Click Commit

CLion confirms with a brief notification showing how many files were committed.

6. Push your branch to GitHub

  1. Click the Git Branch button → Push… (or press ⇧⌘K)
  2. A dialog shows your branch name with an arrow pointing to origin : your-branch-name and the label New — this is correct for a first push of a new branch
  3. Click Push

CLion confirms with "Pushed [branch-name] to new branch origin/[branch-name]". Your branch and its commits are now on GitHub.

You may see a Create pull request link in the confirmation. As a solo developer you can ignore this — you will merge directly in the IDE in the steps below.

7. Switch back to main

  1. Click the Git Branch button
  2. Find main in the list
  3. Click the > arrow to the right of main
  4. Click Checkout

The Git Branch button now shows main. CLion may briefly show a message saying "Workspace associated with branch 'main' has been restored" — this is normal, ignore it.

8. Merge your branch into main

  1. Click the Git Branch button
  2. Find your feature branch in the list
  3. Click the > arrow to the right of it
  4. Click Merge '[your-branch]' into 'main'

CLion confirms with "Merged [your-branch] to main" and offers a Delete [your-branch] link. Click it — the branch has done its job and can be removed.

9. Push main to GitHub

  1. Click the Git Branch button → Push… (or press ⇧⌘K)
  2. Click Push

Your merged changes are now on GitHub. You are back on main, ready to start the next branch whenever you need one.

Steps 5 and 6 (commit and push the branch) can be combined into a single click: in the Commit dialog, use the dropdown arrow next to the Commit button and choose Commit and Push. This saves one step when you know you're ready to push immediately.

Clone a C/C++ Repository

  1. File → New Project from Version Control…
  2. Select GitHub — your repos appear automatically
  3. Click the repo, choose a local directory, click Clone
  4. If the project has a CMakeLists.txt, CLion detects it and offers to configure it as a CMake project — click Yes

Setting Up .gitignore for C/C++ Projects

When you created the repo on GitHub, you had the option to add a .gitignore template. If you chose C++, GitHub already created one that excludes build output. If not, you can add one now:

  1. In the Project panel, right-click your project root → New → File
  2. Name it .gitignore
  3. Add lines for the folders and file types to exclude. At minimum:
  • cmake-build-debug/
  • cmake-build-release/
  • build/
  • .idea/
  • *.o, *.a, *.so, *.dylib, *.exe

Once saved, CLion's Commit panel will no longer show build artifacts as changed files. Stage and commit the .gitignore file itself so teammates get it too.

If .idea/ Was Already Committed

Adding .idea/ to your .gitignore only prevents future commits — it has no effect on files Git is already tracking. If CLion's Commit panel or Git Log shows .idea/ files in earlier commits, you need to explicitly tell Git to stop watching them. Open CLion's built-in terminal (View → Tool Windows → Terminal) and run these two commands: 

git rm -r --cached .idea
git commit -m "Stop tracking .idea folder"

What each command does:

  • git rm -r --cached .idea — removes the entire .idea/ folder from Git's index (what Git is watching). The -r means recursive (the whole folder), and --cached means "only remove it from Git's tracking — leave the actual files on disk untouched." Your IDE settings are not deleted.
  • git commit -m "Stop tracking .idea folder" — saves that removal as a commit, so the untracking becomes part of the repo's history.

After this, push as normal. The .gitignore entry keeps .idea/ out of all future commits permanently — you will never need to run these commands again for this repo.

This fix applies to the current repo on the machine where you run it. Each computer manages its own Git index independently. If you have the same repo cloned on another machine and .idea/ was tracked there too, run the same two commands on that machine.

Preventing This on Every Future Project (Global .gitignore)

Rather than adding .idea/ to the .gitignore of every new project, you can create a single global ignore file that Git applies to every repo on your machine automatically. Open the Terminal and run: 

echo ".idea/" >> ~/.gitignore_global
git config --global core.excludesfile ~/.gitignore_global

The first line adds .idea/ to a file in your home folder. The second tells Git to use that file as a global ignore list for all repos. From this point forward, you never need to think about .idea/ again on this computer — it will never appear as an untracked file in any project. Run the same two lines on your desktop Mac to get the same behaviour there.

Viewing Blame and File History

Git Blame shows which commit (and which person) last touched each line. It's invaluable in C/C++ for understanding why a specific line or decision exists.

  1. Right-click anywhere in the editor → Git → Annotate with Git Blame
  2. A gutter appears on the left showing the author, date, and commit ID for each line
  3. Click any annotation to jump to the full commit details

To see a file's full change history: right-click the file in the Project panel → Git → Show History. A timeline appears showing every commit that touched this file, with full diffs.

Managing Submodules

C/C++ projects often use Git submodules to vendor third-party libraries. CLion can show submodules in the Project panel, but initializing them requires one terminal command — this is the one case where the terminal is unavoidable.

Open View → Terminal inside CLion and run:

git submodule update --init --recursive

After that, CLion treats the submodule's source files like any other part of your project — you can browse, search, and navigate into them normally.

After every Git → Pull, you may need to run the submodule command again if the upstream repo updated a submodule version. CLion doesn't handle this automatically.

Common Pitfalls

GitHub has a reputation for being confusing, and not without reason. It was built by and for people who already understood version control — and it never quite stopped assuming you do too. These are the mistakes almost everyone makes in the first few weeks.

The Three Layers Nobody Explains

The root cause of most GitHub confusion is that there are three separate places your files can live, and actions move things between them:

LayerWhereHow to get there
Your files on diskYour computer — visible in FinderJust editing normally
Your local Git repoYour computer — hidden inside the project folderCommitting
GitHubThe cloudPushing

When something "isn't showing up" somewhere, the answer is almost always that you moved it one layer but not the other. The fix is always the same: figure out which layer it's stuck in and do the next step.

"I committed but my teammate can't see it"

This is the most common mistake of all. Committing saves your work locally. Pushing sends it to GitHub. They are two separate steps, and it's easy to do the first and forget the second — especially because committing feels like saving.

The fix: get into the habit of using Commit and Push… as a single action instead of committing and pushing separately. In Rider and CLion, this is one button in the Commit panel. It does both in one step.

When you sit down to work, Pull first. When you finish work, Commit and Push. If you make those two habits automatic, most GitHub problems disappear entirely.

"I pulled and it says 'already up to date' but I still can't see the file"

This usually means one of two things:

  • The file was committed but never pushed — on the other machine, check that the commit was actually pushed to GitHub, not just saved locally.
  • Rider's project view needs a refresh — the file may be on disk but the IDE hasn't noticed it yet. Right-click the folder in the Project panel and choose Reload from Disk.

"My push was rejected"

This almost always means someone (or another machine) pushed a commit to GitHub since you last pulled. GitHub won't let you push if your local history is behind. The fix is to Pull first, let the IDE merge the changes, then Push again.

Never click "Force Push" when this happens. Force push overwrites the remote history and can permanently erase work. Always pull first.

"I created the repo with a README and now the push is rejected"

When you create a GitHub repo with a README, it already has one commit. When you try to push a local project that has its own commits, GitHub sees two unrelated histories and rejects the push. The simplest fix: delete the README.md from github.com (open the file → trash icon → commit the deletion), making the repo empty, then push from your IDE.

To avoid this entirely: never check "Add a README" when creating a repo that you plan to push local files into. Only use it when starting a brand new project from scratch on GitHub.

"GitHub's website is overwhelming"

It is. GitHub serves solo developers, startups, and enormous engineering organisations all on the same interface. For your day-to-day work, you only ever need three things on the website:

  • Creating a new repository
  • Opening a Pull Request after pushing a branch
  • Merging a Pull Request once it's reviewed

Actions, Issues, Discussions, Projects, Marketplace, Sponsors, Copilot, Pages — ignore all of it until you need it. Most people never do.

Troubleshooting

I committed to main by mistake

If you haven't pushed yet:

In Rider or CLion:

  1. Open the Git Log (Cmd+9 / Alt+9)
  2. Right-click the commit on main → Undo Commit…
  3. Your changes return to the Commit panel as staged changes
  4. Click the Git Branch button in the Git Branch button (top toolbar) (shows the current branch name) → New Branch → name it and click Create
  5. Commit your changes on this new branch

In Xcode: open the built-in terminal (Source Control → Open in Terminal), run git reset --soft HEAD~1, then create a new branch from the Source Control Navigator.

My push was rejected

This usually means someone else pushed to the same branch since you last pulled. Use Git → Pull… first to bring in their changes, resolve any conflicts using the IDE's merge editor, and then push again.

Never use "force push" on a shared branch like main. It rewrites history and can permanently erase teammates' work.

I have changes but need to switch branches urgently

Use Stash (described in the Rider and Xcode sections above) to set your changes aside without committing, switch branches, do the urgent work, then come back and apply the stash.

I can't see a teammate's branch

Rider/CLion: Go to Git → Fetch to download the latest branch list from GitHub, then look for the branch under Remote Branches in the branch switcher.
Xcode: Go to Source Control → Fetch Changes, then expand your remote in the Source Control Navigator.

There's a file I never want Git to track

Add it to your .gitignore file. Right-click the file in the Project panel → Git → Add to .gitignore (Rider/CLion) — this adds the entry automatically. In Xcode, open your .gitignore and add the filename or pattern manually.

I deleted a file and need it back

Rider/CLion: Open the Git Log, find the last commit that contained the file, right-click it → Checkout Revision… → navigate to the file → Checkout.
Xcode: In the Source Control Navigator, open the branch history, find the relevant commit, and use Checkout from the context menu.

Good Habits from Day One

  • Commit small and often. A commit that does one thing is easy to review, revert, and understand six months later. Use your IDE's Commit panel after every meaningful unit of work.
  • Write meaningful commit messages. "Fix bug" is useless. "Fix crash when user logs out with no active session" is useful.
  • Pull before you start new work. Switch to main, pull, then create your branch. Starting from stale code is the most common cause of merge conflicts.
  • Never commit secrets. API keys, passwords, and tokens must never go into a repo. Store them in environment variables or a .env file, and add that file to .gitignore before your first commit.
  • Use .gitignore from day one. Choose a template when creating your GitHub repo. Your IDE will stop flagging build artifacts as untracked files.
  • Branch for everything. Even solo, feature branches mean your main is always in a working, working state — safe to show anyone.
  • Review your diff before every commit. Your IDE shows exactly what changed. A 30-second review before committing catches far more mistakes than code review after the fact.
Create a throwaway repo called git-sandbox on GitHub and experiment freely. Clone it into each IDE, make branches, create conflicts, undo commits. Breaking things in a sandbox is the fastest way to build real confidence.
