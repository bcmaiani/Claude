MusicKit

A complete beginner's tutorial for integrating Apple Music into an iOS app — organized around classical music

Last updated: 2026-07-01

This guide walks through everything a complete beginner needs to add Apple Music search and playback to a real iOS app using MusicKit for Swift. Unlike most MusicKit tutorials — which are written around pop/rock songs and albums — every step here is built for classical repertoire: composers, works, and movements. MusicKit's data model was designed with popular music in mind, so classical apps need a few extra layers on top of the framework. This doc is built to hand you those layers directly, not bolt them on as an afterthought.

Xcode 16 or later

MusicKit is built into the SDK — no extra package to install.

iOS 15.0+ deployment target

Core APIs shipped in iOS 15; classical-relevant filtering needs iOS 16+.

Paid Apple Developer Program

A free/personal-team account cannot enable the MusicKit capability.

Apple Music subscription (for testing)

Full classical recordings need a subscription; otherwise 30-second previews only.

MusicKit's data model does not have first-class support for classical music the way the separate Apple Music Classical app does. There is no Composer type, no Period/era, and no conductor, orchestra, or opus-number fields anywhere in the framework. The Classical Music in MusicKit section below covers exactly what you get, what you don't, and how to build the missing layer yourself.

What Is MusicKit?

MusicKit is Apple's Swift framework for integrating apps with Apple Music. It's built with Swift concurrency (async/await) and SwiftUI in mind, and it runs on iOS, iPadOS, macOS, tvOS, visionOS, and watchOS.

With MusicKit, once a user grants permission, your app can:

  • Search and browse the full Apple Music catalog (songs, albums, artists, playlists, stations)
  • Read and modify the user's personal music library
  • Create playlists and add songs to the user's library
  • Play catalog and library content through a built-in player, with full queue control

MusicKit was introduced at WWDC 2021 as "MusicKit for Swift" — a native replacement for having to hand-roll calls to the Apple Music web API and manage your own JWT-based developer tokens. Its type system (Song, Album, Artist, Playlist, Genre, and so on) was clearly designed around the shape of popular music catalogs — a fact that becomes relevant the moment you start working with symphonies, concertos, and multi-movement works.

Getting Set Up

Two one-time steps happen outside your Swift code: enabling the MusicKit capability on your App ID, and adding a usage-description string so iOS can show the permission prompt.

Enable the MusicKit Capability

  1. Sign in at developer.apple.com/account with a paid Developer Program membership.
  2. Go to Certificates, Identifiers & Profiles → Identifiers.
  3. Select your app's Bundle ID (or click + to register a new App ID if you haven't yet).
  4. Under App Services (a.k.a. Capabilities), check the box for MusicKit, then save.
  5. Back in Xcode, select your project → your target → Signing & Capabilities → + Capability → add MusicKit. Xcode will sync the capability from your App ID.
With MusicKit for Swift, once the capability is enabled, the developer token and user token are generated and managed for you automatically. You never touch a private key or hand-build a JWT — that manual process only applies to MusicKit JS and the server-to-server Apple Music API.

Add an Info.plist Usage String

iOS requires an explanation string before it will show the Apple Music permission prompt. Add this key to your Info.plist (or the Info tab of your target settings):

Info.plist
<key>NSAppleMusicUsageDescription</key>
<string>This app uses your Apple Music account to search for and play classical recordings.</string>
If this key is missing, your app will crash the moment it tries to request authorization or access MusicKit — this is one of the most common first-run mistakes.

Classical Music in MusicKit

This is the part most MusicKit tutorials skip. Read it before you start writing search or playback code — it changes how you'll model everything downstream.

MusicKit's core types — Song, Album, Artist — map cleanly onto a pop/rock catalog: one artist, one song, one album, done. Classical music doesn't fit that shape. A single recording can involve a composer (long dead), a performing ensemble, a conductor, one or more soloists, a specific work with an opus or catalog number, and several movements that are really "one piece of music" split across multiple Song entries. MusicKit gives you a handful of properties that gesture at this reality, but it does not model it as a first-class concept anywhere in the framework.

Classical-Specific Song Properties

Song (verified directly against Apple's current MusicKit reference) exposes five properties explicitly documented as "For classical music only":

PropertyTypePurpose
workNameString?The name of the associated work (e.g. "Symphony No. 5 in C Minor")
movementNameString?The movement's name (e.g. "II. Andante con moto")
movementNumberInt?The movement's position within the work
movementCountInt?Total number of movements in the work
attributionString?The artist or composer to credit for the song

Two more properties aren't classical-exclusive but are essential for building a composer-first UI:

PropertyTypePurpose
composerNameString?A flat display string of the composer(s)
composersMusicItemCollection<Artist>?Structured composer data — reuses the Artist type (see below)
All of the fields above are populated only when Apple's catalog metadata includes them — coverage varies by recording, and non-classical content will simply return nil. Always code defensively around these being optional.

What MusicKit Doesn't Model Gap

This table was verified directly against the full Song and Album property lists in Apple's MusicKit reference — nothing here is a guess:

MissingWhat you get instead
A Composer typeComposers are represented as plain Artist structs via composers/composerName — the same type used for performers. Nothing in the type system distinguishes "wrote it" from "performed it."
A Work typeNothing groups the Song entries that make up one work. You get workName/movementNumber as strings/ints on each track and must group them yourself.
Period / era (Baroque, Classical, Romantic, etc.)No property or type exists anywhere in MusicKit for this. If you want era classification, you're building and maintaining that mapping yourself.
ConductorNo dedicated field. In practice, conductor names sometimes appear folded into artistName or attribution as free text — not reliably parseable.
Orchestra / ensembleSame situation as conductor — no dedicated field; may appear in artistName/artists alongside (or instead of) the conductor, with no way to tell them apart programmatically.
Soloist(s)No dedicated field. Soloists typically show up as additional entries in artists, again indistinguishable from the ensemble or conductor by type.
Opus / catalog number (K., BWV, Op., etc.)Not modeled. If present at all, it's embedded as text inside workName or title.

The Album-Level Gap

All five classical-specific properties live on Song — Album has none of them. An Album exposes title, artistName, artists, genres, recordLabels, and a tracks collection, but nothing that says "this album is a complete Beethoven symphony cycle" or "this album contains three different works." To know that, you have to fetch album.tracks and inspect workName/composerName on every individual track.

Filtering the Library by Composer

There is one genuinely useful, verified capability worth calling out: as of iOS 16, LibrarySongFilter — the filter protocol used by MusicLibraryRequest<Song> — includes composerName as a real, structured filter property for the user's library (not the catalog):

Filtering the user's library by composer — iOS 16+
var request = MusicLibraryRequest()
request.filter(matching: \.composerName, equalTo: "Ludwig van Beethoven")
let response = try await request.response()
This filter only applies to MusicLibraryRequest (the user's personal library). MusicCatalogSearchRequest has no composer-specific filter at all — catalog search only accepts a free-text term, so composer-precise catalog search has to be done by post-filtering results client-side (see Step 3).

Apple Music Classical API Limits

Apple's own Apple Music Classical app (launched 2023) has dramatically richer classical metadata — work-centric browsing, conductor, orchestra, soloists, and catalog/opus numbers, with search that understands composer, work, opus number, and nickname together. Developers have asked whether this enriched metadata is reachable through the public MusicKit/Apple Music API. Reports on Apple's developer forums indicate it is not: API calls return the same metadata available through the standard Music app, not the Classical app's enhanced data set. Plan your app around the properties documented in this section — don't assume Classical-app-level detail will show up in your API responses.

Core Concepts

A handful of general MusicKit ideas show up throughout every app, classical or not. Understanding them up front makes the step-by-step build much easier to follow.

Catalog vs. Library

MusicKit draws content from two distinct sources, and most types come in matching pairs:

Apple Music Catalog

  • The full Apple Music service — every available recording
  • Accessed via MusicCatalogSearchRequest, MusicCatalogResourceRequest
  • No composer-specific filter — free-text term only
  • Requires an active subscription for full playback (previews otherwise)

User's Music Library

  • Recordings the user has actually added to their library
  • Accessed via MusicLibraryRequest, MusicLibrarySearchRequest
  • Supports a real composerName filter (iOS 16+)
  • Works even without a subscription, if content was added while subscribed or purchased

MusicItem Types

Everything MusicKit returns conforms to the MusicItem protocol. The main concrete types:

TypeRepresentsClassical relevance
SongA single track — for classical music, usually one movementCarries all classical-specific properties
AlbumA collection of songs released togetherNo classical-specific properties — see the Album-Level Gap
ArtistA musician, band, or other performerAlso used to represent composers — no dedicated Composer type
PlaylistA user-created or curated ordered collection of songsCould be used to model a "work" manually, but is not classical-aware
GenreA genre category, e.g. "Classical"Coarse — one genre per song/album, not era- or period-specific
MusicVideoA video associated with a track—
StationA continuous, non-repeating stream of music—

Authorization Model

MusicKit uses a single system-level permission, separate from your app's own login (if any). You call MusicAuthorization.request(), iOS shows a system prompt, and the user grants or denies access. The result is a MusicAuthorization.Status:

StatusMeaning
.notDeterminedUser hasn't been asked yet
.deniedUser declined — direct them to Settings to change it
.restrictedAccess is restricted by parental controls / device policy
.authorizedAccess granted — MusicKit calls can proceed
Request authorization early — ideally on a dedicated onboarding screen or right before the user's first search — rather than silently in the background. iOS only shows the system prompt once; after that, a denial can only be reversed by the user in Settings.

Subscription Status

MusicSubscription.current tells you what the signed-in Apple ID can actually do. The property you'll check most often is canPlayCatalogContent, which is true only when the account has an active Apple Music subscription that allows full catalog playback (including from third-party apps). If it's false, check canBecomeSubscriber to decide whether to show Apple's built-in subscription offer sheet.

Tokens: Swift vs. JS

It's easy to find outdated tutorials that describe manual token handling. That workflow is for a different, older technology:

MusicKit for Swift (this tutorial)

  • Native iOS/macOS/etc. framework
  • Developer token generated and refreshed automatically
  • User token also handled automatically after authorization
  • You never touch a .p8 private key

MusicKit JS / Apple Music API (web)

  • For websites and server-to-server use
  • You generate a media identifier + private key in the Developer portal
  • You manually sign a JWT developer token
  • Music user tokens expire (~6 months) and need manual refresh logic

Step 1 — Create the Project

  1. In Xcode: File → New → Project → iOS → App.
  2. Give it a name (e.g. ClassicalPlayer), choose SwiftUI for Interface and Swift for Language.
  3. Set the deployment target to iOS 16.0 if you want the composer library filter from Filtering the Library by Composer, or 15.0 if you only need catalog search and playback.
  4. Complete the MusicKit capability and Info.plist steps above for this project.
  5. At the top of any file that uses it, add import MusicKit.

Step 2 — Request Authorization

Create a small observable class to own authorization state. This example uses ObservableObject for broad compatibility back to iOS 15 (the newer @Observable macro is a fine substitute if your deployment target is iOS 17+).

MusicAuthManager.swift
import MusicKit
import Foundation

@MainActor
class MusicAuthManager: ObservableObject {
    @Published var authorizationStatus: MusicAuthorization.Status = .notDetermined

    init() {
        authorizationStatus = MusicAuthorization.currentStatus
    }

    func requestAuthorization() async {
        let status = await MusicAuthorization.request()
        authorizationStatus = status
    }
}

Call it from a view when the user taps a "Connect Apple Music" button:

Requesting from a SwiftUI view
import SwiftUI

struct ConnectView: View {
    @StateObject private var authManager = MusicAuthManager()

    var body: some View {
        Button("Connect Apple Music") {
            Task {
                await authManager.requestAuthorization()
            }
        }
    }
}
@StateObject is what creates and owns the MusicAuthManager instance here. It ensures the same object survives across SwiftUI's view redraws (rather than being recreated from scratch each time), and — because authorizationStatus is marked @Published — automatically re-renders this view whenever that status changes.
Check authorizationStatus before making any catalog or playback call. If it isn't .authorized, show a prompt instead of calling MusicKit — an unauthorized call throws rather than silently failing.

Step 3 — Search by Composer

MusicCatalogSearchRequest only takes a free-text term — as established in Filtering the Library by Composer, there is no composer-specific catalog filter. Searching a composer's name as the term works reasonably well in practice (Apple's catalog search does weight composer/attribution text), but you should still post-filter for precision using composerName:

Searching the catalog by composer
import MusicKit

func searchByComposer(_ composerName: String) async throws -> [Song] {
    var request = MusicCatalogSearchRequest(term: composerName, types: [Song.self])
    request.limit = 25
    let response = try await request.response()

    // MusicCatalogSearchRequest has no composer-specific filter, so narrow
    // results client-side using the actual composerName field.
    return response.songs.filter {
        ($0.composerName ?? "").localizedCaseInsensitiveContains(composerName)
    }
}
Because coverage of composerName varies by recording, an overly strict filter can hide legitimate results. Consider showing unfiltered results with filtered ones ranked first, rather than discarding non-matches outright.

Step 4 — Group Songs into Works

This is the step that doesn't exist in a typical MusicKit tutorial, because pop music doesn't need it. Since MusicKit has no Work type, build a lightweight one yourself and group the flat list of Song results by composer + work name, ordering movements by movementNumber:

Work.swift
import MusicKit

struct Work: Identifiable, Hashable {
    let id: String
    let composerName: String
    let workName: String
    let movements: [Song]
}

func groupIntoWorks(_ songs: [Song]) -> [Work] {
    let byWork = Dictionary(grouping: songs) { song -> String in
        let composer = song.composerName ?? "Unknown Composer"
        let work = song.workName ?? song.albumTitle ?? song.title
        return "\(composer)::\(work)"
    }

    return byWork.map { key, songsInWork -> Work in
        let sorted = songsInWork.sorted {
            ($0.movementNumber ?? 0) < ($1.movementNumber ?? 0)
        }
        let first = sorted.first!
        return Work(
            id: key,
            composerName: first.composerName ?? "Unknown Composer",
            workName: first.workName ?? first.albumTitle ?? first.title,
            movements: sorted
        )
    }
    .sorted { $0.workName < $1.workName }
}
workName and movementNumber are only populated for recordings Apple has tagged as classical, and even then coverage isn't universal. Always fall back gracefully (as above, to albumTitle then title) rather than assuming these fields are present.

Step 5 — Show Works in SwiftUI

Wire search and grouping into a view model, then render works as expandable groups of movements:

ClassicalSearchViewModel.swift
import MusicKit
import Foundation

@MainActor
class ClassicalSearchViewModel: ObservableObject {
    @Published var works: [Work] = []
    @Published var composerSearchTerm = ""

    func search() async {
        guard !composerSearchTerm.isEmpty else { works = []; return }
        do {
            let songs = try await searchByComposer(composerSearchTerm)
            works = groupIntoWorks(songs)
        } catch {
            print("Search failed: \(error)")
            works = []
        }
    }
}
WorksListView.swift
import SwiftUI
import MusicKit

struct WorksListView: View {
    @StateObject private var viewModel = ClassicalSearchViewModel()
    @StateObject private var player = ClassicalPlayerViewModel()

    var body: some View {
        NavigationStack {
            List(viewModel.works) { work in
                DisclosureGroup {
                    ForEach(work.movements) { movement in
                        Button {
                            Task { await player.play(work: work, startingAt: movement) }
                        } label: {
                            HStack {
                                if let number = movement.movementNumber {
                                    Text("\(number).").foregroundStyle(.secondary)
                                }
                                Text(movement.movementName ?? movement.title)
                            }
                        }
                    }
                } label: {
                    VStack(alignment: .leading) {
                        Text(work.workName).font(.headline)
                        Text(work.composerName).font(.subheadline).foregroundStyle(.secondary)
                    }
                }
            }
            .searchable(text: $viewModel.composerSearchTerm, prompt: "Search by composer")
            .onChange(of: viewModel.composerSearchTerm) {
                Task { await viewModel.search() }
            }
            .navigationTitle("Classical Works")
        }
    }
}

Step 6 — Play a Work in Order

ApplicationMusicPlayer.shared is a single, app-wide player instance. Its queue initializer accepts a startingAt: item, which is exactly what's needed to start a work from a specific movement while keeping the rest of the work queued in order:

ClassicalPlayerViewModel.swift
import MusicKit
import Foundation

@MainActor
class ClassicalPlayerViewModel: ObservableObject {
    private let player = ApplicationMusicPlayer.shared
    @Published var isPlaying = false

    /// Plays an entire work in movement order, optionally starting partway through.
    func play(work: Work, startingAt movement: Song? = nil) async {
        do {
            player.queue = ApplicationMusicPlayer.Queue(
                for: work.movements,
                startingAt: movement
            )
            try await player.play()
            isPlaying = true
        } catch {
            print("Playback failed: \(error)")
        }
    }

    /// Advances to the next movement in the current work.
    func nextMovement() async {
        try? await player.skipToNextEntry()
    }

    /// Returns to the previous movement in the current work.
    func previousMovement() async {
        try? await player.skipToPreviousEntry()
    }

    func pause() {
        player.pause()
        isPlaying = false
    }
}
ApplicationMusicPlayer is observable — its state (playback status, shuffle, repeat) and queue can be read directly in SwiftUI views to keep your UI in sync with what's actually playing, including movement changes triggered from Control Center or the Lock Screen.

Step 7 — Filter the User's Library

For content the user has already added to their library, use the real composerName filter from Filtering the Library by Composer instead of the catalog's free-text search — it's precise, not a heuristic:

Library composer search — iOS 16+
func librarySongs(byComposer composerName: String) async throws -> [Work] {
    var request = MusicLibraryRequest()
    request.filter(matching: \.composerName, equalTo: composerName)
    let response = try await request.response()
    return groupIntoWorks(Array(response.items))
}

This reuses the same groupIntoWorks(_:) helper from Step 4 — the work/movement grouping logic doesn't care whether the songs came from the catalog or the library.

Step 8 — Handle No Subscription

Before or during playback, check whether the signed-in account can actually play full catalog content. This matters more for classical apps than most — a 30-second preview rarely covers even the opening of a symphonic movement, so it's worth prompting for a subscription rather than letting playback silently fall back to a preview clip:

Checking subscription status
import MusicKit

func checkSubscription() async -> Bool {
    do {
        let subscription = try await MusicSubscription.current
        return subscription.canPlayCatalogContent
    } catch {
        return false
    }
}
Offering to subscribe (iOS 16+)
struct WorksListView: View {
    @State private var showingSubscriptionOffer = false
    // ...

    var body: some View {
        List { /* ... */ }
            .musicSubscriptionOffer(isPresented: $showingSubscriptionOffer)
    }
}
Without a subscription, Song.previewAssets still gives you a 30-second preview URL that can be played with plain AVFoundation (AVPlayer) — useful for "try before you subscribe," even if it can't showcase a full movement.

Common Pitfalls

  • Assuming workName/movementNumber are always present. They're populated inconsistently even within classical recordings. Design your UI to degrade gracefully to title/albumTitle.
  • Trusting artistName to mean "composer." For classical recordings it often means the performer or ensemble — use composerName/composers specifically for composer data.
  • Expecting a composer filter on catalog search. MusicCatalogSearchRequest has no such filter; only MusicLibraryRequest does (iOS 16+).
  • Grouping works by title text alone. Different recordings can format the same work's title slightly differently. Group by composer + work name together, and expect to do some normalization for a production app.
  • Missing Info.plist key. No NSAppleMusicUsageDescription means an instant crash on the first authorization or MusicKit call.
  • Free Apple Developer account. The MusicKit capability cannot be enabled without a paid Developer Program membership.
  • Assuming Apple Music Classical's rich metadata is available via the API. Conductor, ensemble, opus numbers, and work-centric browsing from the Classical app are not exposed through MusicKit or the standard Apple Music API.
