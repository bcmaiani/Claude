Closures in Swift

A detailed, example-driven tutorial — every closure shown from its most verbose form down to its simplest

Last updated: 2026-07-03

A closure is a self-contained block of functionality that can be passed around and used in your code, much like a function. Closures can capture and store references to any constants and variables from the context in which they're defined — a behavior known as closing over those values, which is where the name comes from.

Every code example below is shown at least twice: once written as verbosely as Swift allows, and again collapsed down through each syntactic shortcut Swift offers, ending at the simplest legal form. Try reading each ladder top to bottom — it doubles as a tour of Swift's type inference and closure sugar.

What Closures Can Do That a Function Can't

Here's the honest version, because Swift's own documentation is upfront about it: a named function you write with the func keyword is, technically, also a closure — the Swift Programming Language guide calls global and nested functions "special cases of closures." So the interesting question isn't "closures vs. functions" as two unrelated things. It's: what can a closure expression (the anonymous { ... } literal form) do that a func declaration structurally cannot?

Five capabilities are unique to closure expressions:

CapabilityClosure expressionfunc declaration
Write it anonymously, inline, at the call site Yes — no name, no separate declaration No — must be declared elsewhere, then referenced by name
Collapse to near-zero syntax (inferred types, implicit return, $0, trailing closure) Yes — a full ladder of shorthands No — a func always needs its keyword, name, and parameter list spelled out
Be returned from a function and keep carrying its captured state Yes — the natural, idiomatic pattern (factories, counters) Only via an equivalent nested function, which is itself just an unnamed-at-the-call-site closure in disguise
Defer evaluation of a plain expression automatically via @autoclosure Yes — caller writes a bare expression; Swift wraps it Not applicable — a function's arguments are always evaluated as ordinary expressions before the call
Opt in/out of outliving the call via @escaping / Sendable / sending Yes — this vocabulary exists specifically for closures A named function reference passed as an argument is treated under these same closure rules, not separate ones

In short: closures aren't a fundamentally different kind of code from functions — they're the same underlying feature (a callable value with a parameter list and a body) plus an entire layer of syntax and semantics for writing that value anonymously, tersely, and with explicit control over its lifetime. That's the gap this tutorial fills in, example by example.

Anatomy of a Closure Expression

The general form of a closure expression:

General form{ (parameters) -> ReturnType in statements }

Compare that to a function declaration. A function is named up front with the func keyword; a closure has no name and no func keyword — the opening brace is the closure, and the in keyword is what separates the parameter/return-type header from the body.

Function declaration

func multiply(_ a: Int, _ b: Int) -> Int {
    return a * b
}

Equivalent closure expression

let multiply = { (a: Int, b: Int) -> Int in
    return a * b
}
Both multiply values above have the exact same type: (Int, Int) -> Int. Swift doesn't distinguish "function type" from "closure type" — a function reference and a closure literal are interchangeable wherever that type is expected.

The Syntax Spectrum: Verbose → Simplest

Worked example: sorting an array of strings in descending order using Array.sorted(by:), whose signature is sorted(by areInIncreasingOrder: (Element, Element) -> Bool) -> [Element]. Every stage below is functionally identical. 
let names = ["Chris", "Alex", "Ewa", "Barry", "Daniella"]
0Baseline — a named function passed by reference

This isn't a closure at all — it's the function-only starting point everything below improves on.

func backward(_ s1: String, _ s2: String) -> Bool {
    return s1 > s2
}

var reversedNames = names.sorted(by: backward)
1Fully verbose closure expression

Explicit parameter types, explicit return type, explicit return, passed by argument label.

reversedNames = names.sorted(by: { (s1: String, s2: String) -> Bool in
    return s1 > s2
})
2Types inferred from context

Swift already knows sorted(by:) needs (String, String) -> Bool, so the parameter and return types can be dropped.

reversedNames = names.sorted(by: { s1, s2 in return s1 > s2 })
3Implicit return

A single-expression closure body can omit the return keyword — the expression's value is returned automatically.

reversedNames = names.sorted(by: { s1, s2 in s1 > s2 })
4Shorthand argument names

Swift automatically provides $0, $1, … for inline closures, letting you drop the parameter list and the in keyword entirely.

reversedNames = names.sorted(by: { $0 > $1 })
5Trailing closure syntax

Since the closure is the function's only (and last) argument, it can be written outside the parentheses — and here the parentheses themselves can be dropped along with the argument label.

reversedNames = names.sorted { $0 > $1 }
6Operator method reference — the simplest legal form

> is itself a function of type (String, String) -> Bool (via Comparable), so it can be passed directly. This isn't a closure literal anymore — it's the same "function as first-class value" idea from the table above, arriving at the simplest possible spelling. 

reversedNames = names.sorted(by: >)
There's no single "correct" stopping point on this ladder. Reach for whichever step is most readable at the call site — sorted { $0 > $1 } is idiomatic Swift; a five-line named function is better when the logic is genuinely complex or reused elsewhere.

Trailing Closure Syntax

If a closure expression is a function's final argument, you can write it after the closing parenthesis of the call — or, if it's the only argument, drop the parentheses altogether.

Setup
func performTask(_ name: String, onSuccess: () -> Void) {
    print("Running \(name)")
    onSuccess()
}
1Normal call syntax
performTask("upload", onSuccess: {
    print("upload finished")
})
2Trailing closure syntax

The argument label onSuccess: is dropped when the closure moves outside the parentheses.

performTask("upload") {
    print("upload finished")
}
Trailing closure syntax only removes the label at the call site — the parameter still has a name (onSuccess) inside the function body and in its declaration. Don't confuse "no label at the call site" with "no name at all."
3When the closure is the only argument, the parentheses disappear too

performTask above always needs parentheses for "upload" — that argument never goes away. But if a function takes nothing but a closure, there's nothing left to keep the parentheses around: 

func showAlert(action: () -> Void) {
    action()
}

showAlert(action: { print("shown") })   // normal call syntax
showAlert() { print("shown") }          // trailing closure, empty parens kept
showAlert { print("shown") }            // trailing closure, parens dropped entirely

All three lines call showAlert identically. The last form is what the sentence above this example was describing — it's only possible because there's no other argument competing for space inside the parentheses.

Why Closures Exist: Control, Not Data

If you look at onSuccess: () -> Void and think "a block of code with no inputs and no output — what is that even for?", that reaction is correct, not a gap in your understanding. In a simple, one-shot, synchronous example it genuinely isn't doing anything for you. The payoff only shows up once the function you're calling controls something you can't: whether, when, or how many times your code runs.

The trivial case: no benefit at all

Take the performTask example from a few sections back. Calling it with a trailing closure and just writing the same two lines yourself produce identical output:

With a closure parameter

func performTask(_ name: String, onSuccess: () -> Void) {
    print("Running \(name)")
    onSuccess()
}

performTask("upload") {
    print("upload finished")
}
// Running upload
// upload finished

No closure at all

print("Running upload")
print("upload finished")

// Running upload
// upload finished

Same two lines print, in the same order, either way. Nothing about closures is required to get this behavior — which means that example, by itself, can't show you why closures matter. It's only useful for showing syntax. Two changes make the closure version stop being optional.

Case 1 — the function decides how many times

Here, the caller hands over a piece of code and the function decides how often to run it. There's no way to write "the same thing" by just placing code after the call — you'd need to know the repeat count yourself and write a loop, which defeats the point of calling a shared retry function at all. 

func retry(times: Int, _ operation: () -> Void) {
    for _ in 1...times {
        operation()
    }
}

retry(times: 3) {
    print("attempt")
}
// attempt
// attempt
// attempt

retry doesn't know or care what "attempt" means — printing, a network call, anything. It only owns the looping decision. The closure is how you hand it code without also handing it your loop.

Case 2 — the function decides when

This is the case that shows up constantly in real apps: work that doesn't finish by the time the function returns — a network call, a user tapping a button, anything that has to wait for something outside your program's own code. To keep this focused on closures rather than on any particular concurrency API, here's a minimal stand-in: a function that saves your closure instead of running it, and a separate function that runs it whenever it's eventually called. 

class TaskRunner {
    private var pending: (() -> Void)?

    func performTaskLater(_ name: String, onSuccess: @escaping () -> Void) {
        print("Scheduled \(name)")
        pending = onSuccess   // stored, not called — runs later, if at all
    }

    func finishPendingTask() {
        pending?()   // *now* your closure actually runs
        pending = nil
    }
}

let runner = TaskRunner()

runner.performTaskLater("upload") {
    print("upload finished")
}
print("This runs immediately — performTaskLater already returned")

// ... arbitrarily later, e.g. once a real network response arrives ...
runner.finishPendingTask()

// Scheduled upload
// This runs immediately — performTaskLater already returned
// upload finished

performTaskLater returns immediately after printing "Scheduled upload" — it does not run your closure. Your { print("upload finished") } sits inside pending, doing nothing, until something else entirely — a later, separate call to finishPendingTask() — decides to trigger it. There's no way to reproduce that by just writing code after the call; the code you wrote genuinely doesn't run until TaskRunner chooses to run it.

Notice the parameter also needed @escaping: onSuccess: () -> Void would be an error here. By default Swift assumes a closure argument is used and discarded before its function returns; the moment you store it somewhere (like the pending property above) so it can run after the function has already returned, you have to mark it @escaping to tell Swift — and anyone reading the signature — that this closure's lifetime isn't tied to the function call at all. (There's a full section on @escaping later in this doc.)

Line-by-line walkthrough of the TaskRunner example

The whole example above, broken down one line at a time — useful if any single piece of syntax (the optional closure property, @escaping, optional chaining on a call, trailing closure syntax) needs to be isolated from the rest.

1Optional closure property
private var pending: (() -> Void)?

A private, mutable property that can either hold a closure of type () -> Void, or be nil. private restricts read/write access to code inside TaskRunner; var makes it reassignable over time; the trailing ? makes the property itself optional — it may hold a closure, or hold nothing.

2Method signature, with @escaping
func performTaskLater(_ name: String, onSuccess: @escaping () -> Void) {

Declares a method taking an unlabeled String (name — the underscore drops the argument label at the call site) and a labeled, escaping closure (onSuccess) of type () -> Void. @escaping means this method is allowed to store onSuccess and call it after the method itself has already returned.

3Log statement
print("Scheduled \(name)")

Prints a message using string interpolation — \(name) evaluates the parameter and inserts its value into the string. Nothing closure-related happens on this line; it just confirms the method was called, before anything gets stored.

4Storing the closure without calling it
pending = onSuccess   // stored, not called — runs later, if at all

onSuccess with no parentheses refers to the closure itself, without running it — the difference between referring to a closure and calling it. This is also the exact line that requires @escaping: it lets the closure persist inside the object after performTaskLater returns, which is precisely what "escaping" means. After this line, the closure is saved but nothing has executed yet.

5End of performTaskLater
}

Closes the method body. Control returns to the caller — note that onSuccess has been stored but never called, so the closure has not run by the time this method finishes.

6Second method declaration
func finishPendingTask() {

A plain method declaration taking no parameters and returning nothing.

7Optional chaining on a call
pending?()   // *now* your closure actually runs

pending?() is optional chaining applied to a function call: if pending currently holds a closure, call it; if it's nil, skip the call entirely with no crash. (Contrast with force-calling via pending!(), which would crash the program if pending were nil.) This is where the closure stored back in performTaskLater — potentially much earlier, from a different call site — finally executes.

8Clearing the stored closure
pending = nil

Resets pending after the call attempt, making each stored closure "one-shot" — a second call to finishPendingTask() with nothing pending just does nothing. It also releases the reference to the closure (and anything it captured), letting ARC reclaim that memory once nothing else holds it.

9End of finishPendingTask, and of the class
    }
}

Closes finishPendingTask, then closes the TaskRunner class itself.

10Creating an instance
let runner = TaskRunner()

Creates an instance of the class. let means runner will always refer to this same object (though the object's own pending property can still change). TaskRunner() calls the compiler-generated default initializer — no arguments needed, since pending already defaults to nil as an optional. After this line, runner.pending is nil.

11Calling performTaskLater with a trailing closure
runner.performTaskLater("upload") {
    print("upload finished")
}

Trailing closure syntax — equivalent to runner.performTaskLater("upload", onSuccess: { print("upload finished") }). Tracing through it: this prints Scheduled upload, stores the closure into pending, and returns — upload finished has not printed yet.

12Code after the call runs immediately
print("This runs immediately — performTaskLater already returned")

A plain print statement with no closures involved. It runs right after performTaskLater returns, because that method stored the closure and exited rather than blocking until the closure ran — the observable proof that storing a closure and calling it are two separate events in time.

13Comment marking a narrative time gap
// ... arbitrarily later, e.g. once a real network response arrives ...

Not executable code — it signals, narratively, that some unspecified amount of time (or some other real-world event, like a network callback) is meant to have passed before the next line. Nothing in the sample enforces that gap, but in a real program finishPendingTask() would typically be invoked from an entirely different context, such as a network library's own completion handler.

14Finally triggering the stored closure
runner.finishPendingTask()

Calls finishPendingTask(). Inside: pending?() sees that pending holds the closure from step 11 and runs it, printing upload finished; then pending is set back to nil. This produces the full three-line output shown in the code block above.

What if finishPendingTask() were called a second time right after this sequence? By then pending is already nil, so pending?() sees nil and does nothing — no print, no crash. It's a silent no-op, which is exactly the safety optional chaining buys you: with a force-unwrap (pending!()) instead, the second call would crash. This also shows the design is intentionally one-shot — a closure passed into performTaskLater is consumed exactly once; supporting repeat invocation would need a different design (e.g., not niling out pending, or storing closures in a collection instead of a single slot).
That's the whole idea in one sentence: a closure parameter lets the function you're calling — not you — decide if, when, and how many times your code runs. No parameters or return value are needed for that to be useful, because the thing being communicated isn't data, it's a moment in time. Every () -> Void closure you'll see in real Swift code — button actions, animation completions, network callbacks — is this same shape, just wired up to a "when" that's more interesting than a two-second timer.

Multiple Trailing Closures Swift 5.3+

When a function takes more than one closure argument, each additional trailing closure keeps its label.

Setup
func performTask(
    _ name: String,
    onSuccess: () -> Void,
    onFailure: (Error) -> Void
) {
    print("Running \(name)")
    onSuccess()
}
1Normal call syntax — every argument labeled inside the parentheses
performTask("upload", onSuccess: {
    print("done")
}, onFailure: { error in
    print("failed: \(error)")
})
2Multiple trailing closures — first closure's label still dropped, later ones keep theirs
performTask("upload") {
    print("done")
} onFailure: { error in
    print("failed: \(error)")
}

Worked Example: map

let numbers = [1, 2, 3, 4, 5]
1Fully verbose
let doubled = numbers.map({ (n: Int) -> Int in
    return n * 2
})
2Types inferred + implicit return
let doubled2 = numbers.map({ n in n * 2 })
3Shorthand argument name
let doubled3 = numbers.map({ $0 * 2 })
4Trailing closure
let doubled4 = numbers.map { $0 * 2 }

Worked Example: filter

1Fully verbose
let evens = numbers.filter({ (n: Int) -> Bool in
    return n % 2 == 0
})
2Types inferred + implicit return
let evens2 = numbers.filter({ n in n % 2 == 0 })
3Shorthand argument name
let evens3 = numbers.filter({ $0 % 2 == 0 })
4Trailing closure
let evens4 = numbers.filter { $0 % 2 == 0 }

Worked Example: reduce

1Fully verbose
let sum = numbers.reduce(0, { (runningTotal: Int, n: Int) -> Int in
    return runningTotal + n
})
2Types inferred + implicit return
let sum2 = numbers.reduce(0, { runningTotal, n in runningTotal + n })
3Shorthand argument names
let sum3 = numbers.reduce(0, { $0 + $1 })
4Trailing closure
let sum4 = numbers.reduce(0) { $0 + $1 }
5Operator reference — the simplest legal form

+ is itself a function of type (Int, Int) -> Int, so it can be passed directly, same trick as sorted(by: >) earlier.

let sum5 = numbers.reduce(0, +)

Capturing Values

This is the capability table's second row made concrete: a closure can capture a variable from its enclosing scope and carry a live, mutable reference to it — even after that enclosing scope has finished executing.
Classic example — a stateful counter factory
func makeIncrementer(incrementAmount: Int) -> () -> Int {
    var total = 0
    let incrementer: () -> Int = {
        total += incrementAmount   // captures and mutates `total` by reference
        return total
    }
    return incrementer
}

let incrementByTwo = makeIncrementer(incrementAmount: 2)
incrementByTwo()   // 2
incrementByTwo()   // 4
incrementByTwo()   // 6

let incrementByTen = makeIncrementer(incrementAmount: 10)
incrementByTen()   // 10 — an entirely separate captured `total`

total is a local variable inside makeIncrementer. Normally it would be destroyed the moment the function returns. Because the closure captures a reference to it, Swift keeps it alive for as long as the closure exists — one total per call to makeIncrementer, each independent of the others.

A nested func can do this too (func incrementer() -> Int { total += incrementAmount; return total } would behave identically). That's the "special case of a closure" point from the overview: the capturing behavior itself isn't closure-exclusive — what's exclusive is being able to write incrementer as an anonymous, inline { } literal instead of a separate, named declaration.

"Why Not Just Pass It as a Parameter?"

Capturing and passing a parameter often look interchangeable — for reading a value once, they are. The gap opens up in two specific situations, both worth checking for whenever a capture feels like it "should" just be a parameter instead.

1. You don't control the call. Something else decides what gets passed in.

let taxRate = 0.08

func applyTax(_ price: Double, rate: Double) -> Double {
    return price * (1 + rate)
}

prices.map(applyTax)   // ❌ won't compile — map always calls its closure with
                        // exactly ONE argument (the element). There's no call
                        // site where `rate` could be handed over, because
                        // `map`, not you, writes that call.

let withTax = prices.map { $0 * (1 + taxRate) }   // ✅ captures instead of being told

map, filter, sorted, a completion handler, a button's tap handler, Task { } — in every one of these, some other piece of code owns the call and has already fixed how many arguments it will pass. Capturing is the only way for your code to bring in extra context that caller was never going to supply.

2. You want memory across separate calls.

Start with a plain function and nothing else:

func increment(_ current: Int) -> Int {
    return current + 1
}

increment(0)   // 1
increment(0)   // 1 — same input, same output, every single time.
               // The function has no idea it was ever called before.

To make this look like it's counting up, the caller has to remember the last result and feed it back in by hand, every time: 

var current = 0
current = increment(current)   // 1
current = increment(current)   // 2
current = increment(current)   // 3

This counts, but the memory is current — sitting out in the caller's code, not in increment. The function itself is exactly as forgetful as it was on the first line. Now compare a closure that captures its own counter instead of being handed one: 

func makeCounter() -> () -> Int {
    var current = 0
    return {
        current += 1
        return current
    }
}

let counter = makeCounter()
counter()   // 1
counter()   // 2
counter()   // 3

counter() takes zero arguments — there is no slot for anyone to hand a value back in, even if they wanted to. It still counts up, because current isn't a parameter at all here: it's a variable living inside the closure, created once when makeCounter() ran, and kept alive for as long as counter itself exists. Nobody outside is tracking it.

A parameter's value always comes from the caller, supplied fresh on every single call. A captured variable's value lives inside the closure itself, so a call that takes no arguments at all — like counter() — can still "remember" what happened last time.

Capture Lists

By default, a closure captures variables by reference — it always sees the current value. A capture list lets you override that, snapshotting a value type at the moment the closure is created, or controlling reference-counting behavior for reference types.

1Default capture — by reference (sees later mutations)
var x = 10
let printXByReference = { print(x) }
x = 20
printXByReference()   // prints 20 — sees the mutation
2Capture list — snapshot at creation time
var y = 10
let printXSnapshot = { [y] in print(y) }
y = 20
printXSnapshot()   // prints 10 — captured the value at closure-creation time

The same syntax controls reference types, most commonly to avoid retain cycles with self:

Full spectrum of self-capture, most retain-prone to safest / most explicit
class NetworkManager {
    var onComplete: (() -> Void)?

    func fetch() {
        // 1 — Strong capture (default): risks a reference cycle if `self`
        //     also holds a strong reference back to this closure.
        onComplete = {
            self.updateUI()
        }

        // 2 — [weak self]: self becomes Optional inside the closure.
        onComplete = { [weak self] in
            self?.updateUI()
        }

        // 3 — [weak self] with an early-exit guard, when you need
        //     several uses of self in the body.
        onComplete = { [weak self] in
            guard let self else { return }
            self.updateUI()
        }

        // 4 — [unowned self]: no Optional unwrapping, but crashes
        //     if self has been deallocated by the time this runs.
        //     Only use when self is guaranteed to outlive the closure.
        onComplete = { [unowned self] in
            self.updateUI()
        }
    }

    func updateUI() { /* ... */ }
}
[unowned self] trades safety for slightly less ceremony. If there's any chance the closure runs after self is deallocated, prefer [weak self] — a crash from a dangling unowned reference is much harder to debug than an optional you forgot to unwrap.

Closures Are Reference Types

Assigning a closure to a new constant or variable doesn't copy its captured state — both names point at the same underlying closure and share its captures. 

let incrementByTwo = makeIncrementer(incrementAmount: 2)
incrementByTwo()   // 2

let alsoIncrementByTwo = incrementByTwo   // reference copy, not a fresh counter
alsoIncrementByTwo()   // 4 — shares state with incrementByTwo
incrementByTwo()       // 6 — still the same shared `total`

@escaping Closures

A closure parameter is non-escaping by default — Swift assumes it will be called and discarded before the function returns. Mark it @escaping when it needs to outlive the call, e.g. because it's stored in a property or invoked later on another queue. This is the same TaskRunner example from the "Why Closures Exist" section — here it is again, laid out as a direct non-escaping-vs-escaping comparison.
1Non-escaping (default) — this will not compile if you try to store `onSuccess`
class TaskRunner {
    // ❌ Compile error: "escaping closure captures non-escaping parameter"
    // private var pending: (() -> Void)?
    // func performTaskLater(_ name: String, onSuccess: () -> Void) {
    //     pending = onSuccess
    // }
}
2Marked @escaping — now it's legal to store and call later
class TaskRunner {
    private var pending: (() -> Void)?

    func performTaskLater(_ name: String, onSuccess: @escaping () -> Void) {
        print("Scheduled \(name)")
        pending = onSuccess   // stored, not called — runs later, if at all
    }

    func finishPendingTask() {
        pending?()   // *now* your closure actually runs
        pending = nil
    }
}

let runner = TaskRunner()

runner.performTaskLater("upload") {
    print("upload finished")
}
print("This runs immediately — performTaskLater already returned")

// ... arbitrarily later, e.g. once a real network response arrives ...
runner.finishPendingTask()

// Scheduled upload
// This runs immediately — performTaskLater already returned
// upload finished

performTaskLater returns immediately after printing "Scheduled upload" — it does not run your closure. Your { print("upload finished") } sits inside pending, doing nothing, until something else entirely — a later, separate call to finishPendingTask() — decides to trigger it. There's no way to reproduce that by just writing code after the call; the code you wrote genuinely doesn't run until TaskRunner chooses to run it.

That's exactly why the parameter needs @escaping: onSuccess: () -> Void would be an error here. By default Swift assumes a closure argument is used and discarded before its function returns; the moment you store it somewhere (like the pending property above) so it can run after the function has already returned, you have to mark it @escaping to tell Swift — and anyone reading the signature — that this closure's lifetime isn't tied to the function call at all.

3A more realistic signature, then simplified with a typealias
// Verbose inline closure type
func fetch(completion: @escaping (Result) -> Void) { /* ... */ }

// With a typealias — the @escaping attribute still applies at the call site
typealias FetchCompletion = (Result) -> Void
func fetch(completion: @escaping FetchCompletion) { /* ... */ }
@escaping itself hasn't gotten any less necessary — any parameter a function stores, returns, or hands to another asynchronous API still needs it. What changed in Swift 5.3 (SE-0269) is narrower: inside an @escaping closure you no longer have to write self. explicitly when self is a value type, or when you've already listed self in the capture list — Swift can tell a reference cycle isn't possible in those two cases.

@autoclosure

@autoclosure automatically wraps a plain expression argument in a closure, so the call site can look like an ordinary function call with no braces at all — while the callee still controls exactly when (or whether) that expression is evaluated.
Setup — a logging helper
func logIfTrue(_ predicate: () -> Bool) {
    if predicate() {
        print("Condition was true")
    }
}
1Without @autoclosure — caller must write an explicit closure
logIfTrue({ 2 > 1 })
logIfTrue { 2 > 1 }   // trailing closure syntax still applies
2With @autoclosure — caller writes a bare expression
func logIfTrue(_ predicate: @autoclosure () -> Bool) {
    if predicate() {
        print("Condition was true")
    }
}

logIfTrue(2 > 1)   // no braces — reads like a plain function call
3Combined with @escaping, for deferred evaluation
func logLater(_ predicate: @autoclosure @escaping () -> Bool) {
    DispatchQueue.main.async {
        if predicate() { print("Condition was true, evaluated later") }
    }
}
This is exactly how ?? and assert(_:_:) work in the standard library: the right-hand side (or the assertion message) is only evaluated if it's actually needed, which matters when that expression is expensive or has side effects.
@autoclosure hides the fact that a closure is being created at every call site. Apple's own guidance is to use it sparingly — it can make code that looks like a simple expression secretly defer or repeat work, which is surprising to readers.

Closures as Types

Closures have a real type — (ParameterTypes) -> ReturnType — and can be used anywhere a type is expected: as a property, a typealias, an array element, or a dictionary value.

typealias for a closure type
typealias MathOperation = (Int, Int) -> Int

let add: MathOperation = { $0 + $1 }
let multiply: MathOperation = { $0 * $1 }
Stored in a dictionary and invoked dynamically
let operations: [String: MathOperation] = [
    "+": add,
    "*": multiply
]

operations["+"]?(3, 4)   // 7
operations["*"]?(3, 4)   // 12
As a default parameter value
func combine(_ a: Int, _ b: Int, using operation: MathOperation = { $0 + $1 }) -> Int {
    operation(a, b)
}

combine(2, 3)             // 5  — uses the default (+)
combine(2, 3, using: multiply)   // 6
Optional closure property
class Button {
    var onTap: (() -> Void)?
}

let button = Button()
button.onTap = { print("Tapped!") }
button.onTap?()   // safe to call even if nil

Sendable & sending Closures Swift 6

Swift 6's strict concurrency checking adds a closure-specific vocabulary for controlling which isolation domain (thread, actor, task) a closure and its captured values are allowed to cross into.
@Sendable — closure and its captures must be safe to share across domains
func fetchData(completion: @escaping @Sendable (Data) -> Void) {
    // `completion` may run on a different thread/actor than the caller
}
sending — transfers exclusive ownership of a non-Sendable capture
func process(_ work: sending @escaping () -> Void) {
    // the caller gives up its own references to whatever `work` captured
}
Unlike @unchecked Sendable, sending doesn't bypass the compiler's data-race checks — it changes which checks apply, verifying that ownership of the captured values transfers cleanly instead of requiring every capture to be Sendable.

Common Pitfalls

Retain cycles. A class instance holding a closure property, whose closure strongly captures self, creates a cycle — neither can be deallocated. Use [weak self] or [unowned self] whenever a closure is stored (not just called and discarded).
Confusing "trailing" with "unnamed." Trailing closure syntax only removes the label at the call site. The parameter still has whatever internal name it was declared with — autocomplete and documentation will still show it.
Value types vs. reference types in captures. Capturing a struct or other value type by reference (the default, no capture list) still means the closure sees later mutations to that variable, because the variable itself is captured, not a snapshot of its value at that instant. Only an explicit capture list ({ [x] in ... }) snapshots it.
