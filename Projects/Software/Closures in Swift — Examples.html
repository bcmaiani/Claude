Closures in Swift — Examples

A living gallery of copiously annotated closure examples, organized by what they're used for rather than by syntax

Last updated: 2026-07-06

This doc is a companion to Closures in Swift — Living Doc, which covers syntax, capture semantics, and attributes in tutorial form. This one skips the tutorial and goes straight to real-shaped code: one example after another, each with a plain-English annotation and — the point of the whole exercise — an explicit answer to "could a func do this instead?"

That question has three honest answers, and every example below is tagged with one:

Requires a closure — capturing local state, deferring evaluation, or escaping into a stored/async lifetime. A func structurally cannot do this; there's no workaround that isn't itself a closure in disguise.

Func + workaround — possible with a plain func, but only by promoting local state to a global, a stored property, or an extra boolean flag — moving something that wanted to be scoped to one call site out into the open for the whole program to see.

Func works fine — the closure is purely a convenience here. Included on purpose, because pretending every closure is load-bearing would be dishonest. Recognizing this case is just as useful as recognizing the other two.

The short version of "control, not data": a closure's payload usually isn't the values it takes or returns — it's when, how many times, and with what captured context it runs, none of which shows up in its type signature. Watch for that thread running through nearly every "requires a closure" verdict below.

Collection Operations

map — basic transform Func works fine

let prices = [12.99, 5.50, 22.00, 8.75]

func applyStandardTax(_ price: Double) -> Double {
    return price * 1.08
}

let withTaxViaFunc    = prices.map(applyStandardTax)
let withTaxViaClosure = prices.map { $0 * 1.08 }
// Both produce [14.0292, 5.94, 23.76, 9.45]

map only needs a value of type (Double) -> Double — it doesn't care whether that value is a named function or an anonymous closure literal, because in Swift they're the same underlying type.

A named func works identically here. The closure's only advantage is not having to invent and declare a name (applyStandardTax) for a one-off calculation used exactly once.

map — capturing a runtime rate Requires a closure

let taxRate = 0.08   // scoped to right here, nothing more — no parameter, no global
let withTax = prices.map { $0 * (1 + taxRate) }   // ...yet the closure sees it, simply
                                                    // because it's written in this same scope

The closure reaches directly into the enclosing scope and reads taxRate — no parameter, no global, just a local constant that happens to live one line above.

A top-level func applyTax(_ price: Double) -> Double has no way to see taxRate at all — a top-level function has no enclosing scope to capture in the first place. (A nested func, declared inside another function, could see taxRate too — nested funcs capture their enclosing scope exactly like closure literals do. What's unique here is writing the logic anonymously, inline, at the call site.) The nearest func-only fix that keeps things top-level is currying: 
func makeTaxApplier(rate: Double) -> (Double) -> Double {
    return { price in price * (1 + rate) }   // ← still a closure, just returned from a func
}
prices.map(makeTaxApplier(rate: taxRate))
which doesn't actually avoid closures — it just relocates one into a return value. Once behavior needs to be parameterized by a value known only at runtime, something has to close over that value, full stop.
"Couldn't I just pass taxRate in as a second parameter?" No — and the reason is that you don't write the call to your own closure here, map does. map's implementation always invokes its closure with exactly one argument, the element: prices.map(applyTax) fails to compile the moment applyTax needs a second rate parameter, because there is no call site — inside map, buried in the standard library — where you could hand rate over. Capturing sidesteps that entirely: the closure already knows taxRate before map ever calls it, so it doesn't need to be told.

filter — capturing a threshold Requires a closure

let scores = [55, 82, 91, 40, 76, 88]
let passingScore = 60   // could come from a config file, a user setting, anything non-literal

let passed = scores.filter { $0 >= passingScore }
// [82, 91, 76, 88]
Same trap as the tax example: filter, not you, decides how many arguments your closure receives — exactly one, the element. A top-level func isPassing(_ score: Int, threshold: Int) -> Bool can't be handed to filter at all, because filter has no idea it's supposed to supply threshold. "Pass it in explicitly" was never actually available at this call site — only capturing passingScore, or promoting it to global scope, are real options, and the global route isn't a shorter-vs-longer tradeoff: it makes the value visible to everything else in the file or module, drops any automatic cleanup, and breaks the moment two call sites want their own independent passingScore at once.

reduce(into:) — word frequency count Func works fine

let words = "the cat sat on the mat the cat ran".split(separator: " ").map(String.init)

let frequencies = words.reduce(into: [String: Int]()) { counts, word in
    counts[word, default: 0] += 1
}
// ["the": 3, "cat": 2, "sat": 1, "on": 1, "mat": 1, "ran": 1]

reduce(into:) hands the closure an inout-like accumulator it mutates directly, instead of returning a brand-new value every iteration the way plain reduce(_:_:) does — cheaper for a growing dictionary, since nothing gets copied on each step.

This closure captures nothing from the outside — it only touches its own two parameters. A named func tally(_ counts: inout [String: Int], _ word: String) plugs into reduce(into:) exactly as well. Closures win here purely on "don't make me name and declare this separately."

sorted — multi-key comparator Func works fine

struct Player { let name: String; let score: Int }

let players = [
    Player(name: "Nadia", score: 91),
    Player(name: "Omar",  score: 91),
    Player(name: "Priya", score: 78)
]

let ranked = players.sorted {
    if $0.score != $1.score { return $0.score > $1.score }   // higher score first
    return $0.name < $1.name                                 // tie-break alphabetically
}
// Nadia (91), Omar (91), Priya (78)
Multi-branch but self-contained — this comparator captures nothing external. A named func rankPlayers(_ a: Player, _ b: Player) -> Bool would work identically and is arguably more readable once the tie-break logic grows past a line or two. Complexity alone isn't a reason to prefer a closure; capturing state is.

forEach — mutating an outer variable Requires a closure

var total = 0
[1, 2, 3, 4].forEach { total += $0 }
// total == 10
The closure mutates total, a var declared one line above, by reference. Swift's own language guide calls named functions "special cases of closures" — and a nested func inside the same scope could do this too — but a genuinely top-level, separately-declared func has no access to total whatsoever. There's no parameter you could add to a free-standing func that would let it reach back into this specific call site's local variable.

Completion Handlers & Escaping Callbacks

Escaping completion handler with capture Requires a closure

struct User { let id: Int; let name: String }
struct APIClient {
    func fetchUser(id: Int, completion: @escaping (Result<User, Error>) -> Void) {
        // ... kicks off a network request; completion runs later, off this call stack ...
    }
}

let client = APIClient()
var requestCount = 0

client.fetchUser(id: 42) { result in
    requestCount += 1
    switch result {
    case .success(let user): print("Got \(user.name) — request #\(requestCount)")
    case .failure(let error): print("Failed: \(error)")
    }
}
The closure captures requestCount and mutates it whenever the response eventually arrives — possibly on a different thread, possibly seconds later. A top-level func has no enclosing scope to reach back into; the only func-only fix is promoting requestCount to a global or an instance property, which changes its lifetime and visibility for the entire program just to serve one call site.

Bridging a callback API to async/await Requires a closure

func fetchUser(id: Int) async throws -> User {
    try await withCheckedThrowingContinuation { continuation in
        client.fetchUser(id: id) { result in
            continuation.resume(with: result)
        }
    }
}
Two nested closures, each capturing something from just outside it: the outer one captures continuation so the inner completion handler can resume it exactly once. This bridging pattern — write an anonymous block right at the call site, capturing a value that exists nowhere else — is precisely what @escaping closures were designed for; a named func would need its own dedicated continuation-holding property, recreating a class just to replace one closure.

Stored Closures as Properties

onTap-style property Requires a closure

final class Button {
    var onTap: (() -> Void)?
    func simulateTap() { onTap?() }
}

let saveButton = Button()
var saveCount = 0

saveButton.onTap = {
    saveCount += 1
    print("Saved \(saveCount) time(s)")
}

saveButton.simulateTap()   // Saved 1 time(s)
saveButton.simulateTap()   // Saved 2 time(s)
Button itself has no idea saveCount exists — it lives in the scope where the button was configured, not inside the class. You could assign a named func reference to onTap too, but that func could never see this particular saveCount; only a closure literal written at this exact call site can reach into this exact local variable.

lazy var + immediately-invoked closure Func + workaround

class ReportGenerator {
    let data: [Int]

    lazy var summary: String = {
        let total = data.reduce(0, +)
        let average = Double(total) / Double(data.count)
        return "Total: \(total), Average: \(average)"
    }()

    init(data: [Int]) { self.data = data }
}

The trailing () calls the closure immediately, but lazy means that call only actually happens the first time summary is read — and the result is cached from then on.

A func computeSummary() could compute the same string, but you'd lose the automatic once-only caching — you'd have to hand-roll it with a private Bool flag and a backing var: 
private var _summary: String?
func summary() -> String {
    if let cached = _summary { return cached }
    let total = data.reduce(0, +)
    let result = "Total: \(total)"
    _summary = result
    return result
}
Workable, but it's the closure-based lazy var automating exactly this pattern for you — one line instead of five.

State Capture: Factories & Memoization

Every example in this section could raise "why not just pass the value in as a parameter?" — and here the answer isn't about who controls the call (as with map earlier), it's that a parameter never has memory. A parameter is only ever what the caller hands over, fresh, on that one call — it can't remember anything from a previous call unless the caller tracks it and passes it back in themselves, every time. A capture gives the closure its own private storage that persists silently between separate calls, with no caller involved in keeping it up to date.

Stateful ID generator factory Requires a closure

func makeIDGenerator(prefix: String) -> () -> String {
    var count = 0
    return {
        count += 1
        return "\(prefix)-\(count)"
    }
}

let userIDGen = makeIDGenerator(prefix: "USR")
userIDGen()   // "USR-1"
userIDGen()   // "USR-2"

let orderIDGen = makeIDGenerator(prefix: "ORD")
orderIDGen()   // "ORD-1" — a completely independent counter
Each call to makeIDGenerator creates a fresh, private count that only the returned closure can see or mutate — two generators never share state, even though they came from the same function. A func cannot be "returned with private memory attached"; the closure is that memory plus the code that touches it, bundled as one value.

Generic memoization cache Requires a closure

func memoize<Input: Hashable, Output>(_ fn: @escaping (Input) -> Output) -> (Input) -> Output {
    var cache: [Input: Output] = [:]
    return { input in
        if let cached = cache[input] { return cached }
        let result = fn(input)
        cache[input] = result
        return result
    }
}

func slowSquare(_ n: Int) -> Int {
    print("Computing...")
    return n * n
}

let fastSquare = memoize(slowSquare)
fastSquare(5)   // prints "Computing...", returns 25
fastSquare(5)   // returns 25 immediately — no print, cache hit
This is "control, not data" in its purest form: fastSquare's type signature, (Int) -> Int, is identical to slowSquare's. Nothing in the signature reveals that one of them remembers previous calls. The difference lives entirely in the hidden cache dictionary the closure captured — a func has no mechanism to carry private, persistent memory like this between separate calls.

Debouncing rapid input Requires a closure

final class Debouncer {
    private var pendingWork: DispatchWorkItem?
    private let delay: TimeInterval
    init(delay: TimeInterval) { self.delay = delay }

    func run(_ action: @escaping () -> Void) {
        pendingWork?.cancel()
        let work = DispatchWorkItem(block: action)
        pendingWork = work
        DispatchQueue.main.asyncAfter(deadline: .now() + delay, execute: work)
    }
}

let searchDebouncer = Debouncer(delay: 0.3)

func handleSearchInput(_ text: String) {
    searchDebouncer.run {
        print("Searching for \(text)")   // captures this specific keystroke's text
    }
}
Every keystroke hands run a fresh closure capturing that keystroke's own text, and the previous pending closure gets cancelled before it ever runs. Debouncer decides if and when your code actually executes — which is the whole "control, not data" idea again: the closure's inputs and outputs (() -> Void) are trivial; what matters is that its execution is entirely out of your hands once you call run.

Custom Control-Flow Functions

retry(times:) Func works fine (mechanically)

func retry<T>(times: Int, operation: () throws -> T) rethrows -> T {
    for attempt in 1..<times {
        if let result = try? operation() { return result }
        print("Attempt \(attempt) failed, retrying...")
    }
    return try operation()   // final attempt — let it throw if it fails
}

let value = try retry(times: 3) {
    try riskyNetworkCall()   // in practice, this closure usually captures local context
}
The signature itself doesn't require capturing anything — a func matching () throws -> T plugs in fine. Marked "func works fine" mechanically, but flagged here because real call sites almost always want to close over something local (a URL, a request ID, a token) — the moment they do, this tips straight into "requires a closure" the same way the map/filter examples did.

measure(_:) — timing helper Requires a closure

func measure<T>(_ label: String, _ work: () -> T) -> T {
    let start = Date()
    let result = work()
    let elapsed = Date().timeIntervalSince(start)
    print("\(label) took \(elapsed)s")
    return result
}

let sortedBigArray = measure("sort") {
    bigArray.sorted()
}
work gets sandwiched between two lines of code that measure owns — the start timestamp and the elapsed-time calculation. A func can't be "inserted between two statements belonging to someone else's implementation"; you'd have to copy-paste the timing boilerplate around every single call instead of writing it once, here, and reusing it.

Transaction / automatic rollback Func + workaround

struct InsufficientFundsError: Error {}

final class Account {
    private(set) var balance: Int
    init(balance: Int) { self.balance = balance }

    func transaction(_ changes: (inout Int) throws -> Void) rethrows {
        var snapshot = balance
        try changes(&snapshot)
        balance = snapshot   // only reached if `changes` never threw
    }
}

let account = Account(balance: 100)
try? account.transaction { balance in
    balance -= 30
    guard balance >= 0 else { throw InsufficientFundsError() }
    balance -= 10
}
The closure receives a mutable snapshot, not the real balance — if it throws partway through, balance is never reassigned, so the whole transaction rolls back for free. A named func matching (inout Int) throws -> Void would work mechanically the same way (no capture is strictly required), which is why this is tagged "workaround" rather than "required" — but in practice the changes you want to apply almost always come from local variables (a withdrawal amount typed by a user, say), and the instant they do, only a capturing closure will do.

Autoclosures

Custom require(_:_:) Requires a closure

func require(_ condition: @autoclosure () -> Bool, _ message: @autoclosure () -> String) {
    if !condition() {
        fatalError(message())
    }
}

require(user.age >= 18, "User \(user.name) must be an adult")
message only gets evaluated — meaning the string interpolation only actually runs — if condition() is false. Passing a plain, eager String parameter (the only func-equivalent alternative) would build that string on every call, passing or failing, which defeats the purpose when the message is expensive to construct or has side effects. @autoclosure only exists because closures can defer evaluation; a function's arguments are always evaluated before the call, with no exception.

Lazy nil-coalescing helper Requires a closure

func firstNonNil<T>(_ a: T?, _ b: @autoclosure () -> T) -> T {
    return a ?? b()
}

let name = firstNonNil(cachedName, expensiveNetworkLookup())
Same idea as Swift's own ?? operator: expensiveNetworkLookup() is only actually called if cachedName is nil. Without @autoclosure, both arguments would be fully evaluated before firstNonNil even started running — there's no way to write "only run this if needed" using a plain function parameter.

Closures as First-Class Values

Dictionary of interchangeable strategies Func + workaround

enum Operation: String { case add, subtract, multiply, divide }

let strategies: [Operation: (Double, Double) -> Double] = [
    .add: (+),
    .subtract: (-),
    .multiply: (*),
    .divide: { numerator, denominator in
        guard denominator != 0 else { return .nan }
        return numerator / denominator
    }
]

func calculate(_ a: Double, _ op: Operation, _ b: Double) -> Double {
    strategies[op]!(a, b)
}

calculate(10, .divide, 2)   // 5.0
Notice (+), (-), and (*) are operator functions slotted directly into the dictionary — proof that funcs and closures share one type and are freely interchangeable as values. The .divide case needed an actual closure literal only because its logic (the zero-guard) isn't an existing named operator. Tagged "workaround" because you could write a private named func for divide too — the dictionary itself doesn't require closures, just callable values, and named funcs qualify.

Function composition operator Requires a closure

infix operator >>>: AdditionPrecedence

func >>> <A, B, C>(_ f: @escaping (A) -> B, _ g: @escaping (B) -> C) -> (A) -> C {
    return { x in g(f(x)) }
}

let trim: (String) -> String      = { $0.trimmingCharacters(in: .whitespaces) }
let lowercase: (String) -> String = { $0.lowercased() }
let normalize = trim >>> lowercase

normalize("  Hello World  ")   // "hello world"
>>> builds a brand-new function value at runtime by capturing f and g inside a fresh closure — something only possible because closures can be manufactured programmatically, inside another function's body, out of other function values. trim and lowercase could individually be named funcs, but "glue two funcs into a third, new one, on the fly" has no func-only equivalent — the glue itself has to be a closure, because it needs to hold onto f and g as captured state.

Concurrency

Task { } — capturing into a concurrent context Requires a closure

func loadDashboard() {
    let userID = currentUser.id

    Task {
        let profile = try? await api.fetchProfile(userID)
        await MainActor.run {
            self.profileLabel.text = profile?.name
        }
    }
}
Task { } takes an async closure that captures userID and self from the synchronous scope around it, then runs concurrently, possibly on a different thread. A synchronous func has no equivalent concept — you cannot "schedule code to run concurrently, later, with today's local values baked in" without packaging that code and its captured context together, which is exactly what a closure is.

Sendable, sending, and weak let Requires a closure Swift 6.3–6.4, 2026

Swift's strict concurrency checking added a vocabulary that applies specifically to closures (and the values they capture), because a closure is a bundle of code plus mutable captured state — and the compiler needs to know whether that whole bundle is safe to move across threads or actors. Plain data has no equivalent problem to solve.

Before Swift 6.3 — a weak var property forces @unchecked Sendable
final class Session: @unchecked Sendable {
    weak var delegate: SessionDelegate?
    // `@unchecked` was required because a mutable `weak var` can't be proven
    // safe by the compiler's own Sendable checking.
}
Swift 6.3+ — weak let is immutable, so real Sendable checking applies
final class Session: Sendable {
    weak let delegate: SessionDelegate?
    // Immutable, so the compiler can now verify Sendable safety
    // without the `@unchecked` escape hatch.
}
sending — transferring ownership of a non-Sendable capture into a closure
actor Downloader {
    func process(_ handler: sending @escaping (Data) -> Void) {
        // The caller gives up its own references to whatever `handler`
        // captured, so this actor can safely take ownership of it.
    }
}
@Sendable, sending, and the new weak let / ~Sendable refinements exist only in relation to closures and the reference types they capture — there's no "Sendable function" problem distinct from "Sendable closure," because a captureless top-level func referenced by name has nothing for these checks to verify. This entire category of compiler vocabulary is proof, at the type-system level, that a closure is fundamentally code bundled with state — and crossing concurrency domains means reasoning about both halves of that bundle at once.

Recursive Closures — Where a Func Is Actually Simpler

Every example so far has shown closures doing something a plain func can't, or can only awkwardly replicate. Recursion flips that: it's a case where a closure needs a workaround, and a named func gets the feature for free — purely because a func has a name it can refer to inside its own body, and a closure literal, by design, does not. 
// A func recurses trivially, by name:
func factorial(_ n: Int) -> Int {
    n <= 1 ? 1 : n * factorial(n - 1)
}
factorial(5)   // 120

Recursive closure via a wrapper type Func + workaround (here, func is simpler)

struct Recursive<T> { let call: (Recursive<T>, Int) -> T }

let factorialClosure = Recursive<Int> { fact, n in
    n <= 1 ? 1 : n * fact.call(fact, n - 1)
}

factorialClosure.call(factorialClosure, 5)   // 120
The closure has to be handed a reference to itself as a parameter on every call, because it has no name it could use to call itself directly from inside its own body — that's the entire reason the Recursive wrapper exists. This is the one spot in this whole doc where the plain func factorial above is unambiguously less code and less ceremony than any closure-based version.

Recursive closure via a self-referencing var Func + workaround

var factorialVar: ((Int) -> Int)!
factorialVar = { n in
    n <= 1 ? 1 : n * factorialVar(n - 1)
}

factorialVar(5)   // 120
This works because factorialVar captures itself by reference — but only because it was declared as an implicitly-unwrapped optional var first, in a separate statement, before being assigned. A func gets to refer to its own name the moment it's declared, in one step; a closure has to manufacture a name for itself in two steps first. Neither closure-based approach here is an improvement on the plain func — they're included specifically to keep this doc's claims honest instead of overselling closures as strictly superior in every case.
