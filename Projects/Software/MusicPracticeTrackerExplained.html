MusicPracticeTrackerExplained

A living translation guide — Swift/SwiftUI/SwiftData explained for a C#/Java background

Last updated: 2026-07-02

This doc exists because Swift keeps not behaving like C# or Java even when the syntax looks close enough to lull you into thinking it will. Every section below is anchored to real code from the MusicPracticeTracker project — nothing here is a generic tutorial example, it's all things the app actually does.

How to use this doc: it's meant to grow. When you ask about a specific line and the explanation is broadly useful (not just "what does this one variable do"), it gets added here as its own section. Treat it as the accumulated FAQ for this codebase, organized by concept rather than by file.

Where's the Controller?

The biggest reorientation isn't a syntax thing, it's structural. In a typical C#/Java MVC or MVVM app, there's a Controller (or ViewModel) class that owns the UI lifecycle: it fetches data, mutates fields, and the View reads from it or gets told to refresh. SwiftUI doesn't have that layer at all, by design.

Instead, a SwiftUI View is a function of its current state: View = f(state). You describe what the UI should look like given the current data, and SwiftUI figures out what changed and re-renders only that. There's no explicit "refresh the view" call anywhere in this codebase — you won't find one, because it isn't a thing you do manually.

C# / WPF-ish mental model

class SessionController {
    void OnDeleteClicked() {
        _repository.Delete(session);
        RefreshView();   // explicit
    }
}

Swift / SwiftUI actual model

Button("Delete") {
    modelContext.delete(session)
    // no refresh call — SwiftUI observes
    // that `sessions` (a @Query) changed
    // and recomputes body automatically
}

Concretely in this project: SessionListView's body reads sessions (a @Query, more on that below). When you delete a session, SwiftData's storage changes, the @Query notices, and SwiftUI reruns body to produce a new description of the list — automatically, with no controller mediating it. The closest analogue in your background is probably data-binding in WPF/XAML, except SwiftUI takes that idea and makes it the entire UI framework, not an opt-in feature.

If you're looking for "where does the app decide what to do when I tap a button" — that logic lives directly in the View, as a closure passed to the button (see "Trailing Closures" below). There's no separate controller file to go find. It can feel like logic is "in the wrong place" until this clicks.

SwiftUI's official guidance has shifted over the years, but for an app this size, putting small pieces of logic (like deleteSession(_:) in SessionListView) directly on the View struct is completely idiomatic — not a shortcut we took because the app is small.

Structs vs. Classes (the one that bites hardest)

In C# and Java, you reach for a class by default and a struct is the special case (and Java doesn't even have user-defined value types outside records). Swift inverts that: structs are the default, and you should have a specific reason to reach for a class.

This isn't just a style preference — it changes semantics. A struct is a value type: assigning it or passing it copies the whole thing, like an int or a C# struct. A class is a reference type, exactly like every class you've ever used in C#/Java — assignment shares the same underlying object.

Every SwiftUI View is a struct

struct SessionListView: View {
    @Environment(\.modelContext) private var modelContext
    @Query(sort: \PracticeSession.date, order: .reverse)
    private var sessions: [PracticeSession]

    var body: some View { ... }
}

Every SwiftData model is a class

@Model
final class PracticeSession {
    var date: Date = Date.now
    var instrument: String?
    var sessionTasks: [SessionTask] = []
}

Why views are structs: SwiftUI recreates view structs constantly — every time body re-runs, it's not "updating" the old SessionListView instance, it's throwing it away and building a brand new one from the current state. That's cheap precisely because it's a value type with no heap allocation or reference counting overhead, similar in spirit to how you wouldn't think twice about creating a new C# struct every frame, but you'd think hard about allocating a new class instance every frame.

Two more advantages beyond raw creation speed:

  • No shared mutable state. Structs are copied, not aliased — there's no way for two parts of the app to hold a reference to "the same" view and mutate it out from under each other. That's a whole category of bug (and the retain-cycle bookkeeping that comes with reference types) that simply can't happen with view structs.
  • Diffing. SwiftUI's rendering engine works by comparing the newly-produced view struct against the previous one to figure out what actually changed on screen — conceptually similar to React's virtual-DOM diffing. Value-type equality is cheap and well-defined to check; comparing object identity on a class instance wouldn't tell you whether its properties changed underneath.

If you've touched React: body being re-invoked on every state change, returning a fresh disposable description of the UI, is the same shape as a function component re-running and returning new JSX each render. The state that actually persists across renders lives outside that returned description in both cases — @State/@Observable here, useState/hooks there.

Why models are classes: PracticeSession needs reference (identity) semantics — when SessionDetailView holds a PracticeSession and mutates session.instrument, every other view holding a reference to that same session needs to see the change. That's exactly the class/reference behavior you already know from C#/Java; SwiftData models happen to use it because identity matters for a database row.

The gotcha: because View is a value type, you cannot just declare a plain var on it and mutate it the way you'd mutate a field on a C# object — a fresh copy gets created on every re-render, so any state would reset constantly. That's exactly what @State exists to solve (see Property Wrappers below): it tells SwiftUI to keep that particular piece of data alive outside the struct's own lifetime, across re-creations.

Optionals

Java has Optional<T> as an opt-in wrapper type you can choose to use. C# has nullable reference types as a relatively recent, partially-enforced compiler feature. Swift's optional (T?) is neither — it's baked into the type system from day one, and the compiler will not let you accidentally treat a String? as a String. There is no unchecked null anywhere in this codebase; if a value can be absent, its type says so explicitly.

Real example — MusicPracticeTracker/Models/PracticeSession.swift
var instrument: String?   // may or may not have a value
var notes: String?

Compare to timerStartedAt on SessionTask, which uses nil itself as meaningful state (not an error case) — "the timer isn't running" is represented as "there's no start time," not as a separate boolean flag:

Real example — MusicPracticeTracker/Models/SessionTask.swift
var timerStartedAt: Date?

var isRunning: Bool { timerStartedAt != nil }

Three ways optionals get unwrapped in this codebase, all doing the same job C# null-checks or Java Optional.isPresent() would:

SwiftWhat it doesRoughly like
guard let start = timerStartedAt else { return } Early-exit if nil; start is a non-optional value for the rest of the function if (timerStartedAt == null) return; then use the now-known-non-null value
if let instrument = session.instrument, !instrument.isEmpty { ... } Only enter the block if non-nil; scoped to the if if (instrument != null && !instrument.isEmpty()) { ... }
session.instrument ?? "" Nil-coalescing — use the right side if left is nil C# ?? — identical operator, same meaning
Real example — MusicPracticeTracker/Views/SessionDetailView.swift
TextField("Instrument (optional)", text: Binding(
    get: { session.instrument ?? "" },
    set: { session.instrument = $0.isEmpty ? nil : $0 }
))
You'll also see ! (force-unwrap) and try! in Swift code out in the wild — they mean "I'm certain this isn't nil / won't throw, crash the app if I'm wrong." This codebase avoids them everywhere except the one deliberate case in MusicPracticeTrackerApp.swift (fatalError on store creation failure, which is intentionally a hard crash during development — see the comment there).

Computed Properties (a "field" that's actually a method)

This one trips people up reading the code before it trips people up writing it. In Swift, var foo: T { ... } with a body but no stored backing looks exactly like a field, but it's a method that runs every time you access it — equivalent to a C# property with only a get accessor, except Swift makes this the idiomatic default everywhere, not a special "property" construct layered on top of fields.

C#

public TimeInterval TotalTimeSpent
{
    get
    {
        return sessionTasks.Sum(t => t.CurrentElapsed);
    }
}

Swift — MusicPracticeTracker/Models/PracticeSession.swift

var totalTimeSpent: TimeInterval {
    sessionTasks.reduce(0) { $0 + $1.currentElapsed }
}

Note also: no return keyword. A single-expression closure or computed-property body implicitly returns its value — this shows up constantly in Swift and is worth just getting used to rather than fighting.

Another real one, from SessionTask — this is the "am I currently timing, including time since the timer started" calculation used to drive the live-updating stopwatch display:

MusicPracticeTracker/Models/SessionTask.swift
var currentElapsed: TimeInterval {
    guard let start = timerStartedAt else { return timeSpent }
    return timeSpent + Date.now.timeIntervalSince(start)
}
Rule of thumb while reading: if a var declaration has { } with code inside instead of = someValue, it recomputes on every access. It is not cached unless you explicitly cache it.

Closures (a.k.a. Lambdas) — What They Actually Are

Worth slowing down on — "I never really got lambdas in C#" is an extremely normal reaction, since the syntax is compact enough to use by pattern-matching without ever building a mental model of what's happening underneath. Swift closures are the identical underlying concept, so it's worth fixing the C# side first.

Step 1: functions as values

Normally you pass data around — an int, a string, an object reference. A lambda is what lets you pass behavior around the same way: a chunk of code, stored in a variable or handed to a method as an argument, that gets invoked later. C# needed a type to describe "a variable that holds a function" before lambda syntax existed at all — that's a delegate:

C# — before lambdas existed
delegate int Operation(int x, int y);

int Add(int x, int y) => x + y;

Operation op = Add;       // a method, stored as a value
int result = op(3, 4);    // invoked like a function: 7

That already proves functions can be values in C# — but Add had to exist as a separately-declared, named method somewhere else just to be wrapped up and handed to op. A lambda expression removes that requirement: you write the function body inline, with no separate declaration anywhere:

C# — the same thing, as a lambda
Func<int, int, int> add = (x, y) => x + y;
int result = add(3, 4);   // 7 — no `Add` method ever declared

Func<int, int, int> reads as "a function taking two ints, returning an int" — the last type parameter is always the return type. Action<T> is the same idea for a function that returns nothing. (x, y) => x + y is the lambda itself: parameters left of =>, body on the right. Exactly like the Swift single-expression rule from Computed Properties above, a one-expression C# arrow lambda implicitly returns it — no return keyword needed.

Step 2: the part that makes it a closure, not just an anonymous function

So far this is just "an unnamed function." The "closure" part specifically describes what happens when that inline code refers to a variable from outside itself:

C# — the classic closure gotcha
var actions = new List<Action>();
int total = 0;
for (int i = 0; i < 3; i++) {
    actions.Add(() => total += i);   // captures `total` and `i`
}
foreach (var a in actions) a();
Console.WriteLine(total);  // 3

Each () => total += i is a closure: it doesn't just read the value of total at the moment it's written — it captures the variable itself, a live reference, and can read and mutate it later, even though the for loop that created the lambda has long since finished running. That's the literal meaning of "closure": the function closes over the variables it references, bundling them together with the code so the whole package can be handed somewhere else and still work correctly. An ordinary function has no memory between calls; a closure does, specifically because of what it captured.

This is also why closures are described as carrying "their environment" with them. If a closure is passed out of the method that created it — stored in a list, handed to another thread, returned from a function — the variables it captured stay alive too, for as long as the closure itself is alive. Garbage collection (C#) or ARC (Swift) won't reclaim them early, because the closure is still holding a reference.

Same concept, Swift syntax

Swift closures work identically — capture by reference, by default — but the syntax is far more compact, and (per "Trailing Closures & the DSL" below) gets special call-site sugar that makes closures read like a native part of the language rather than something bolted on for callbacks.

C#Swift
Func<int, int, int> add = (x, y) => x + y;let add: (Int, Int) -> Int = { x, y in x + y }
Action<string> log = msg => Console.WriteLine(msg);let log: (String) -> Void = { msg in print(msg) }
no shorthand — parameters always named$0, $1... implicit positional parameter names, e.g. { $0 + $1 }
last argument stays inside the parens: list.Where(x => x > 5)trailing closure — moves outside the parens: list.filter { $0 > 5 }
no equivalent — capture behavior is fixedcapture list [weak self] — explicitly overrides the default (strong) capture

That last row matters for a real example already in this codebase — AudioRecorder.swift's poll loop (see "async/await, Task, and @MainActor" above):

MusicPracticeTracker/Services/AudioRecorder.swift
displayTask = Task { [weak self] in
    while let self, !Task.isCancelled {
        ...
    }
}

That closure references self (to read self.recorder and write self.elapsed). By default, a Swift closure captures self strongly — it keeps the AudioRecorder instance alive for as long as the closure exists. Since AudioRecorder also holds onto this Task (as its displayTask property), a strong capture here would create exactly the retain cycle from the C# closure example above, except permanent: the object keeps the closure alive, the closure keeps the object alive, neither is ever released. [weak self] is Swift's capture list syntax, overriding the default to capture self as a weak (non-owning) reference instead — the same intent as making a field a WeakReference<T> in C# to break an equivalent cycle.

Rule of thumb: any time a Swift closure is stored somewhere long-lived (a property, a Task kept around, a subscription/handler) and it references self, check for [weak self] or [unowned self]. If the closure is short-lived and just runs once then goes away — like a Button's action, or the .reduce { } in Computed Properties — a strong capture is completely fine, since there's nothing left for it to keep alive after it finishes.

Enums with Raw Values (and why Swift enums are more powerful than you're expecting)

Java enums can carry fields; C# enums are just named integers. Swift enums support both raw values (like C#) and per-case associated data (like a tagged union / F# discriminated union), though this codebase only uses the simpler raw-value form:

MusicPracticeTracker/Models/Recording.swift
enum RecordingMediaType: String, Codable {
    case audio
    case video
}

: String here means each case has an implicit String raw value ("audio", "video") — used because SwiftData can't store a raw Swift enum directly, so Recording stores the raw string and exposes a computed property to convert back and forth:

MusicPracticeTracker/Models/Recording.swift
var mediaTypeRaw: String = RecordingMediaType.audio.rawValue

var mediaType: RecordingMediaType {
    get { RecordingMediaType(rawValue: mediaTypeRaw) ?? .audio }
    set { mediaTypeRaw = newValue.rawValue }
}

That's a computed property with both a getter and setter — same idea as the C# property example above, just with two accessors instead of one. RecordingMediaType(rawValue:) returns an optional (parsing a string into an enum can fail if the string doesn't match any case), which is why the fallback ?? .audio is there.

Extensions (bolt methods onto types you don't own)

C# has extension methods; this is the closest Swift feature, but Swift's version goes further — you can add computed properties (not just methods) and even make an existing type conform to a new protocol, entirely from outside that type's original definition. There's no restriction to static methods with a special first-parameter syntax like C#.

MusicPracticeTracker/Services/TimeInterval+Formatting.swift (the whole file)
extension TimeInterval {
    /// "H:MM:SS" or "M:SS" if under an hour.
    var formattedDuration: String {
        let total = Int(self.rounded())
        let hours = total / 3600
        let minutes = (total % 3600) / 60
        let seconds = total % 60
        if hours > 0 {
            return String(format: "%d:%02d:%02d", hours, minutes, seconds)
        }
        return String(format: "%d:%02d", minutes, seconds)
    }
}

TimeInterval is just a type alias for Double — a built-in type Apple owns, not something defined in this codebase. This extension makes anyDouble.formattedDuration valid anywhere in the app after this file is compiled in, which is why you see calls like session.totalTimeSpent.formattedDuration scattered across the View files with no import or qualification needed.

C# — extension method

public static class DoubleExtensions
{
    public static string FormattedDuration(
        this double seconds)
    {
        // ...
    }
}
// usage: mySeconds.FormattedDuration()

Swift — extension

extension TimeInterval {
    var formattedDuration: String {
        // ...
    }
}
// usage: mySeconds.formattedDuration
//        (no parens — it's a property, not a method)

Same pattern, one file over, makes an existing Apple type (RecordingMediaType's enum, in this case) conform to Identifiable after the fact, purely so it can be used with SwiftUI's .sheet(item:) modifier:

MusicPracticeTracker/Views/TaskDetailView.swift
extension RecordingMediaType: Identifiable {
    var id: String { rawValue }
}

private(set) — one modifier, not two accessors

C#'s "public getter, private setter" needs two access modifiers written out (public T Foo { get; private set; }). Swift bundles that specific, common pattern into a single compound modifier:

MusicPracticeTracker/Services/AudioRecorder.swift
private(set) var isRecording = false
private(set) var elapsed: TimeInterval = 0
private(set) var lastError: String?

Readable from anywhere (audioRecorder.isRecording works from the View), writable only from inside AudioRecorder itself. Same intent as the C# pattern, just spelled differently.

Property Wrappers — probably the single most alien thing here

There's genuinely no direct C#/Java equivalent to this one. The closest mental bridge: imagine if a C# attribute like [Autowired] or [JsonIgnore] didn't just add metadata for a framework to read via reflection, but actually rewrote how the property's getter and setter work, at compile time, as a language feature. That's a property wrapper. The @Something prefix isn't documentation or an annotation — it fundamentally changes what reading and writing that variable does.

WrapperWhat it actually doesWhere it's used
@State Persists a value across view re-creations (see Structs vs. Classes above — this is the fix for "structs can't hold onto mutable state") pendingDeleteSessions in SessionListView
@Bindable Lets you get a two-way Binding (via $) to a property on a reference type, so a TextField can both read and write it directly @Bindable var session: PracticeSession in SessionDetailView
@Query A live, auto-updating SwiftData fetch — closest analogue is an EF Core IQueryable that automatically refreshes the UI when the underlying rows change, without you writing any refresh logic @Query(sort: \PracticeSession.date, order: .reverse) private var sessions
@Environment Dependency injection, but ambient/implicit rather than constructor-injected — pulls a shared value down from wherever it was set higher up the view tree @Environment(\.modelContext) private var modelContext
@Observable (macro) Makes a class's properties individually trackable, so SwiftUI only re-renders views that actually read the specific property that changed @Observable final class AudioRecorder
@Model (macro) Turns a plain class into a SwiftData persistence entity — roughly like an EF Core [Table]-mapped class or a JPA @Entity, but the macro also rewrites the class's storage under the hood @Model final class PracticeSession
Real example, three wrappers in six lines — MusicPracticeTracker/Views/SessionListView.swift
struct SessionListView: View {
    @Environment(\.modelContext) private var modelContext
    @Query(sort: \PracticeSession.date, order: .reverse)
    private var sessions: [PracticeSession]

    @State private var pendingDeleteSessions: [PracticeSession]?

Reading it: modelContext is handed down implicitly from the app's .modelContainer(modelContainer) call in MusicPracticeTrackerApp.swift — nothing in SessionListView constructs it or receives it as an init parameter. sessions is always current; when a session is deleted elsewhere, this array updates and the list re-renders with zero manual wiring. pendingDeleteSessions is the one piece of this view's own local, transient state, which is why (and only why) it needs @State — everything else here is either injected or derived from the database.

@Model and @Observable are macros — Swift's equivalent of C# source generators or Java annotation processors, but resolved by the compiler itself rather than a separate build step. @Model final class PracticeSession { var date: Date } expands, behind the scenes, into a much larger class with SwiftData's storage machinery bolted on. You're never meant to look at the expanded code, same as you wouldn't normally inspect what a C# source generator emits — but it's worth knowing it's happening, since it's why a two-line class definition can do so much.

Trailing Closures & the SwiftUI DSL

Swift closures are lambdas — functionally close to C# lambdas / Java lambdas. The part that's genuinely different is trailing closure syntax: when a function's last parameter is a closure, you can move it outside the parentheses, and if it's the only parameter, you can drop the parentheses entirely. This is what makes SwiftUI's declarative style possible — without it, every view body would be drowning in parens.

Without trailing closure syntax

Button(action: { deleteSession(session) },
       label: { Label("Delete", systemImage: "trash") })

As actually written — MusicPracticeTracker/Views/SessionListView.swift

Button(role: .destructive) {
    deleteSession(session)
} label: {
    Label("Delete", systemImage: "trash")
}

That's a button with two trailing closures — the multi-trailing-closure syntax (each labeled) is a more recent Swift addition, but the shape should now read naturally: "when tapped, run this; the label looks like this."

The bigger piece under the hood: var body: some View { List { ... } } isn't ordinary Swift — List's content closure is a @ViewBuilder result builder, a compiler feature that lets a block of statements (that don't look like they return anything) implicitly get collected into a single value. This is the same category of language feature as C#'s LINQ query syntax or Kotlin's builder lambdas: special sugar layered on top of the normal language to make one specific pattern (assembling a tree of views) read declaratively instead of requiring `.Add()` calls everywhere.

MusicPracticeTracker/Views/RootTabView.swift
var body: some View {
    TabView {
        NavigationStack {
            SessionListView()
        }
        .tabItem {
            Label("Practice", systemImage: "music.note")
        }

        NavigationStack {
            HistoryView()
        }
        .tabItem {
            Label("History", systemImage: "chart.bar")
        }
    }
}

Nothing in that closure has a return or a .Add(...) call — the result builder is silently collecting the two NavigationStacks into a single "here are my tabs" value behind the scenes. Also note some View as the return type of body: this is an opaque return type — it means "this returns some concrete type that conforms to View, but I'm not telling you which one, and it's not going to change between calls." It exists because the actual concrete type of a deeply nested SwiftUI view tree is unspeakably long and not something you'd ever want to write out by hand.

Why These Closures Were Actually Necessary

A fair question about any closure in this codebase: could it have just been a method instead? Walking through the real ones already shown above, one pattern holds for every single one but the last: something other than this app's own code decides when to call it. That's the actual reason each one had to be a closure — not a stylistic preference.

The delete button — Button's action closure

Recall from "Trailing Closures & the DSL" — MusicPracticeTracker/Views/SessionListView.swift
Button(role: .destructive) {
    deleteSession(session)
} label: {
    Label("Delete", systemImage: "trash")
}

Button decides when to call this closure — the moment the user taps it, on SwiftUI's own timing, not this view's. It also captures session from the enclosing ForEach loop: the specific PracticeSession for this row. A single method like func deleteThisRow() couldn't work on its own — it would need to know which session to delete, and Button never supplies one as an argument. Capturing session right where the closure is written is the only way to get it in.

Notice that deleteSession(session), inside the closure, is itself an ordinary method call — deleteSession(_:) is a plain method on SessionListView, not a closure. Only the outer boundary handed to Button needs to be a closure; everything inside it is regular code.

The instrument text field — Binding's get/set closures

Recall from "Optionals" — MusicPracticeTracker/Views/SessionDetailView.swift
TextField("Instrument (optional)", text: Binding(
    get: { session.instrument ?? "" },
    set: { session.instrument = $0.isEmpty ? nil : $0 }
))

TextField doesn't know what a PracticeSession is and never will — it only knows it was handed something readable and writable, a Binding<String>. SwiftUI decides when to call get (every re-render) and set (every keystroke). Both closures capture session — specifically this screen's session, from SessionDetailView's own @Bindable var session. A top-level pair of functions has no way to know which session to read or write; capturing is the only channel that gets this view's particular session into code that TextField, not you, will call later.

The recording poll loop — Task's closure

Already covered under "Closures (a.k.a. Lambdas)" and "async/await" above, worth restating the pattern: Swift's concurrency runtime, not AudioRecorder itself, decides when this closure's body runs and resumes after each await Task.sleep. It captures self to read self.recorder and write self.elapsed, and needed the [weak self] capture list specifically because it's stored (as displayTask) rather than run once and discarded — the same "escaping, so watch retain cycles" situation as @escaping closures generally.

The honest exception — reduce's closure

Recall from "Computed Properties" — MusicPracticeTracker/Models/PracticeSession.swift
var totalTimeSpent: TimeInterval {
    sessionTasks.reduce(0) { $0 + $1.currentElapsed }
}

This one doesn't fit the pattern above, and it's worth being straightforward about that rather than forcing it to. The closure captures nothing — it only touches its own two parameters ($0, the running total, and $1, each task). A named function, func addElapsed(_ total: TimeInterval, _ task: SessionTask) -> TimeInterval { total + task.currentElapsed }, passed as sessionTasks.reduce(0, addElapsed), would behave identically. This closure exists purely because writing the logic inline, once, reads better than declaring a function used in exactly one place — a real style choice, not a structural requirement.

Restated for this codebase specifically: a closure was necessary wherever SwiftUI or the concurrency runtime — not MusicPracticeTracker's own code — decides when the call happens, and/or the code needed to reach one specific instance (session, self) that a plain top-level function has no way to see. reduce's closure is the one exception here, and recognizing it as an exception matters just as much as recognizing the other four aren't.

More Closures, From a Full Codebase Scan

The examples above were illustrative, not exhaustive. A full pass over every .swift file in the project turned up several more closures worth walking through individually — each one below teaches something the earlier examples didn't. A number of others turned out to be the same lesson in a different file (another ForEach, another delete button, another Binding); rather than repeat those explanations, they're rounded up in the Appendix at the end of this doc, each one pointing back at whichever section above already covers it.

1. A real completion handler — RecordingCaptureView's onFinish

MusicPracticeTracker/Views/RecordingCaptureView.swift
struct RecordingCaptureView: View {
    let mediaType: RecordingMediaType
    let onFinish: (_ fileName: String, _ duration: TimeInterval) -> Void
    ...
}
Call site — MusicPracticeTracker/Views/TaskDetailView.swift
.sheet(item: $showingRecorder) { mediaType in
    RecordingCaptureView(mediaType: mediaType) { fileName, duration in
        let recording = Recording(mediaType: mediaType, duration: duration, fileName: fileName)
        recording.sessionTask = task
        modelContext.insert(recording)
        task.recordings.append(recording)
    }
}

This is the app's one real match to the "escaping completion handler" pattern from the Closures docs — every other completion-shaped closure in this codebase is one SwiftUI itself hands you (Button's action, Binding's get/set); this is a completion handler MusicPracticeTracker designed on purpose. TaskDetailView doesn't know or care how a recording gets made — audio UI, camera UI, whatever — it only needs to be told "a file with this name and duration exists now." The closure captures task and modelContext from TaskDetailView's own scope: exactly the specific task this screen is showing, and this screen's database connection. Neither of those could be known by RecordingCaptureView itself, or by a plain function — only by a closure written inside TaskDetailView, at the point where it already has both.

On the declaring side, Coordinator's initializer marks its copy @escaping: init(onFinish: @escaping (String, TimeInterval) -> Void, dismiss: DismissAction). That's because Coordinator stores onFinish as a property and calls it later, from a delegate callback — the exact "stored, so it must be allowed to outlive the call" situation @escaping exists for.

2. Called repeatedly, on a schedule it controls — TimelineView

MusicPracticeTracker/Views/TaskDetailView.swift
TimelineView(.periodic(from: .now, by: 1)) { _ in
    Text(task.currentElapsed.formattedDuration)
        .font(.system(size: 44, weight: .semibold, design: .rounded))
        .monospacedDigit()
        .frame(maxWidth: .infinity)
}

TimelineView calls this closure over and over, once a second, for as long as the view is on screen — entirely on its own schedule. Nothing in TaskDetailView ever calls it directly, and nothing in TaskDetailView controls how many times it runs, which is the same "the caller decides how many times" idea as retry(times:) in the Examples doc, just driven by a timer instead of a retry count. It also captures task (a @Bindable property of this view) so it can read task.currentElapsed fresh on every single tick — the parameter this closure receives (_, the current timeline context) is deliberately ignored, because the only thing that actually needs to be current is the captured task.

3. Capturing local context in a plain, non-SwiftUI function — PracticeAnalytics

MusicPracticeTracker/Services/PracticeAnalytics.swift
static func dailyTotals(sessions: [PracticeSession], days: Int, calendar: Calendar = .current, now: Date = .now) -> [DailyTotal] {
    let today = calendar.startOfDay(for: now)
    guard let rangeStart = calendar.date(byAdding: .day, value: -(days - 1), to: today) else { return [] }

    var totals: [Date: TimeInterval] = [:]
    for session in sessions { /* ...fills totals... */ }

    return (0..<days).compactMap { offset -> DailyTotal? in
        guard let day = calendar.date(byAdding: .day, value: offset, to: rangeStart) else { return nil }
        return DailyTotal(day: day, totalSeconds: totals[day] ?? 0)
    }
}

Worth seeing a capturing closure that has nothing to do with SwiftUI or concurrency — this is a plain, synchronous, pure static function, and the same rule from the Closures docs still applies unchanged: compactMap, not dailyTotals, decides how many arguments its closure gets (exactly one, the offset), so calendar, rangeStart, and totals — all local to dailyTotals — can only get into that call by being captured. weekdayAverages, a few lines below in the same file, does the identical thing with its own .map { weekday in ... }, capturing totals and counts.

The same file also has the contrasting case, in suggestion(sessions:calendar:now:): weekdays.max(by: { $0.averageSeconds < $1.averageSeconds }) and the .filter { $0.averageSeconds > 0 }.min(by: { ... }) beside it capture nothing at all — both comparators only touch their own $0/$1. Same file, same author, both tiers side by side: capture where local context is genuinely needed, don't where it isn't.

4. A non-lazy immediately-invoked closure — modelContainer

MusicPracticeTracker/MusicPracticeTrackerApp.swift
let modelContainer: ModelContainer = {
    let schema = Schema([PracticeSession.self, SessionTask.self, Recording.self])
    let configuration = ModelConfiguration(schema: schema, isStoredInMemoryOnly: false, cloudKitDatabase: .none)
    do {
        return try ModelContainer(for: schema, configurations: [configuration])
    } catch {
        fatalError("Failed to create ModelContainer: \(error)")
    }
}()

The trailing () calls this closure immediately, exactly once, the moment modelContainer is initialized. It exists because building a ModelContainer takes several statements (build a schema, build a configuration, try/catch) — but a stored property can only be assigned a single expression. Wrapping the whole setup in { ... }() turns "a short procedure" into one expression that produces the value the let needs.

Two things this deliberately isn't: it isn't a computed property (var modelContainer: ModelContainer { ... } with no trailing ()) — that would rebuild the entire container from scratch every time anything reads modelContainer, which is wrong for something meant to be a single, stable store. And it isn't lazy var either — lazy would defer this setup until the property is first read, but the app needs its container to exist immediately at launch, to hand to .modelContainer(modelContainer) a few lines down. Plain let plus an immediately-invoked closure is the one option that runs the setup exactly once, eagerly, right when the type is initialized.

5. Bare function references — no closure syntax at all

MusicPracticeTracker/Views/SessionListView.swift & SessionDetailView.swift
Button(action: addSession) { Label("New Session", systemImage: "plus") }
Button(action: addTask) { Label("Add Task", systemImage: "plus") }
MusicPracticeTracker/Views/RecordingCaptureView.swift
let seconds = (try? await asset.load(.duration)).map(CMTimeGetSeconds) ?? 0

None of these three lines write { } anywhere, and none of them need to. addSession and addTask are ordinary instance methods (private func addSession() { ... }) referenced by name — Swift treats a method written inside its own type as an implicit self.addSession, which is already a value of type () -> Void, the exact type Button's action: parameter wants. CMTimeGetSeconds is a free C function matching (CMTime) -> Double, handed straight to Optional.map the same way sorted(by: >) hands the > operator straight to sorted in the Examples doc. All three are the same idea: whenever an existing function or method already matches the exact type a closure parameter wants, there's nothing for a closure literal to add.

6. Capturing only what's needed — VideoCaptureView.Coordinator

MusicPracticeTracker/Views/RecordingCaptureView.swift
let onFinish = self.onFinish
let dismiss = self.dismiss
Task {
    let asset = AVURLAsset(url: destinationURL)
    let seconds = (try? await asset.load(.duration)).map(CMTimeGetSeconds) ?? 0
    onFinish(fileName, seconds.isFinite ? seconds : 0)
    dismiss()
}

Look closely at the two lines just before Task { }: onFinish and dismiss get copied out of self into their own local constants before the closure that captures them even exists. Once inside Task { }, only those two local values get captured — self itself never does. That's deliberate: nothing guarantees this Coordinator is still alive by the time the await asset.load(.duration) finishes — it's a delegate object with no strong owner of its own once the picker or sheet goes away. Capturing self directly would raise the same question [weak self] exists to answer elsewhere in this codebase (see AudioRecorder's poll loop, above) — but a plain String and a DismissAction value can't "go away" the way an object can, so pulling just those two values out sidesteps the lifetime question entirely instead of answering it with [weak self].

async/await, Task, and @MainActor

Swift's async/await reads almost identically to C#'s — that part will feel familiar. What's different is the compiler-enforced concurrency-safety checking layered on top (Swift 6's "strict concurrency"), which doesn't have a close C#/Java equivalent — the closest comparison is that it's like the compiler enforcing thread-safety rules that in C#/Java you'd only catch via a race-condition bug in production, or a static analyzer you have to opt into and configure.

MusicPracticeTracker/Services/AudioRecorder.swift
func start() async -> Bool {
    let granted = await AVAudioApplication.requestRecordPermission()
    guard granted else {
        lastError = "Microphone access is required..."
        return false
    }
    // ...
}

@MainActor marks a whole class as confined to the main thread — every property access and method call on it is checked at compile time to make sure it only ever happens from the main thread (the UI thread). This is the direct fix for a real error we hit while building this app:

What actually happened: AudioRecorder is an @Observable class with mutable properties (isRecording, elapsed). Xcode's build failed with: "Stored property '_isRecording' of 'Sendable'-conforming class 'AudioRecorder' is mutable." The fix was adding @MainActor above the class — once every access is provably confined to one thread, the compiler is satisfied there's no possibility of a race.
MusicPracticeTracker/Services/AudioRecorder.swift
@MainActor
@Observable
final class AudioRecorder: NSObject, AVAudioRecorderDelegate {
    private(set) var isRecording = false
    ...

The other concurrency piece in this file is a background poll loop that used to be a Timer, and got rewritten as a Task for the same underlying reason:

MusicPracticeTracker/Services/AudioRecorder.swift
private func startDisplayTimer() {
    displayTask?.cancel()
    displayTask = Task { [weak self] in
        while let self, !Task.isCancelled {
            try? await Task.sleep(for: .seconds(0.2))
            guard !Task.isCancelled, let recorder = self.recorder else { return }
            self.elapsed = recorder.currentTime
        }
    }
}

Timer.scheduledTimer's closure runs on the run loop, outside of any actor's isolation — the compiler can't prove it's safe to touch @MainActor state from inside it. A Task { } created from inside an @MainActor method, by contrast, inherits that isolation, so it's allowed to read and write elapsed directly. [weak self] is the same weak-reference pattern you'd use in C# to avoid a retain cycle keeping an object alive forever (Swift uses reference counting, like a simplified, mostly-automatic version of C#'s garbage collector combined with explicit `IDisposable`-style cleanup awareness — more on that if you want a dedicated section later).

Sendable — a compiler-checked "this is thread-safe" promise

Sendable is a marker protocol: a type conforms to it as a claim that instances can be safely passed between threads/tasks without data races. There's no C#/Java equivalent that's enforced by the compiler — you might write a code comment saying "this class is thread-safe" in C#, but nothing checks that claim. In Swift 6, the compiler actually verifies it (or, in the @unchecked case, takes your word for it and shifts the responsibility onto you).

What actually happened: RecordingFileManager is a singleton (static let shared) accessed from multiple places, including background contexts. The build failed with: "Static property 'shared' is not concurrency-safe because non-'Sendable' type 'RecordingFileManager'..." — Swift 6 won't let a mutable-looking singleton be freely shared across threads without proof it's safe.
MusicPracticeTracker/Services/RecordingFileManager.swift
final class RecordingFileManager: @unchecked Sendable {
    static let shared = RecordingFileManager()

    private let directory: URL   // 'let' — set once in init, never reassigned
    ...

@unchecked Sendable means "I'm asserting this myself; don't verify it." That's justified here specifically because every stored property is a let (immutable after init) and every method just reads/writes files on disk with no shared mutable in-memory state — there's genuinely nothing for two threads to race over. If a future version added a mutable cache or counter to this class, that assertion would become a lie, and it'd need a proper fix (an actor, most likely) instead.
