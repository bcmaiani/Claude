The mindset shift, up front

A few things will keep tripping you up if you don't internalize them early:

  1. JavaScript is dynamically typed at runtime, always. TypeScript's types are erased entirely when your code runs — they exist only to help WebStorm and the compiler catch mistakes before runtime. There is no type information left at 2am when something throws in production.
  2. There is no single compiler / build step by default. Node runs JavaScript directly. TypeScript files (.ts) are transpiled to JavaScript before Node ever sees them — WebStorm and modern tooling mostly hide this from you, but it's happening.
  3. The ecosystem is the language. Unlike the .NET BCL or the Java standard library, most of what you'll use day to day comes from npm packages, not a built-in standard library. package.json is your project file; npm is your package manager (think NuGet/Maven, but more central to daily life).
  4. "Interface" doesn't mean what it means in C#/Java. TypeScript interfaces describe the shape of data, not a contract a class explicitly opts into (though classes can opt in too). More on this in Part 3.

Part 1 — Modern JavaScript refresher

You know JS syntax from ~2016 and earlier. Here's what actually changed since then that matters day to day.

Variables: let / const, forget var

var is function-scoped and hoists in confusing ways. Modern JS uses const by default and let when a variable needs reassignment. Both are block-scoped, like C#/Java locals.

const MAX_RETRIES = 3;   // cannot be reassigned
let attempts = 0;        // can be reassigned
attempts += 1;
⚠️ Watch outconst means the binding can't be reassigned, not that the value is immutable. const arr = [1,2,3]; arr.push(4); is legal — you're mutating the array, not reassigning the variable.

Arrow functions & lexical this

An arrow function is just a shorter syntax for writing a function, using => instead of the function keyword. It's one of the most common things you'll see in modern JS, so it's worth being fluent reading it even before the this nuance below.

// Old-style function expression
const add = function(a, b) {
  return a + b;
};

// Same thing, as an arrow function
const add2 = (a, b) => {
  return a + b;
};

// If the body is a single "return" expression, you can drop the braces and "return"
const add3 = (a, b) => a + b;

// One parameter doesn't need parentheses (though many teams keep them for consistency)
const double = x => x * 2;

// No parameters still needs empty parentheses
const sayHi = () => console.log("hi");
🔷 C#/Java comparisonAn arrow function is JavaScript's version of a C# lambda expression ((a, b) => a + b — yes, literally the same arrow) or a Java lambda ((a, b) -> a + b). If you've used LINQ with lambdas or Java's Comparator/Runnable with lambdas, this is the exact same idea: a compact, inline way to write a function, often passed as an argument to another function.
✅ Quick definition: callbackA callback is just a function you pass as an argument to another function, to be called later — either immediately inside that function (like .map() calling it once per array element) or after some delay/event (like a network response or a button click). "Callback" describes the role a function is playing, not a different kind of function — any regular function or arrow function can be used as one.

You'll see them constantly as callbacks passed to other functions, like the array methods below:

const nums = [1, 2, 3];
const doubled = nums.map(n => n * 2);   // pass an arrow function directly into .map()

Beyond the shorter syntax, arrow functions also behave differently with respect to this — that's the part below.

Arrow functions don't have their own this — they capture it from the enclosing scope. This solves the classic "callback loses this" problem you may remember fighting with .bind() in old JS.

class Timer {
  seconds = 0;
  start() {
    // arrow function keeps "this" bound to the Timer instance
    setInterval(() => {
      this.seconds++;
    }, 1000);
  }
}

Use arrow functions for callbacks and short expressions. Regular function declarations are still fine for top-level functions and methods where you want your own this/arguments.

Arrow functions vs. Swift closures

Since you've studied Swift closures carefully, that vocabulary maps onto JS arrow functions almost directly — closer than the C#/Java lambda comparison above. Here's the same terminology, side by side.

Swift closure conceptJS arrow function equivalent
{ (a: Int, b: Int) -> Int in a + b }(a, b) => a + b — no type annotations needed in plain JS; in TypeScript, types are usually inferred from context the same way Swift infers them
$0, $1 shorthand argument namesNo equivalent — JS always names its parameters (n => n * 2, never an unnamed placeholder)
Trailing closure syntax (names.sorted { $0 > $1 })No equivalent syntax — the arrow function is always passed as a normal argument inside parentheses: names.sort((a, b) => b.localeCompare(a))
Capturing a variable "by reference" from the enclosing scopeIdentical behavior, no special term for it — it's just called a closure, same word, same default semantics
Capture list, e.g. { [y] in print(y) } to snapshot a value at creation timeNo equivalent. JS has no capture-list syntax — if you need a snapshot instead of a live reference, you copy the value into a new variable yourself (see example below)
[weak self] / [unowned self] to avoid retain cyclesNo equivalent, and not needed — JS has garbage collection, not ARC/reference counting, so there's no retain-cycle concept for closures to manage
@escaping annotationNo equivalent. Every JS closure behaves like an escaping Swift closure by default — none of them are restricted to "must run before the outer function returns"
numbers.map { $0 * 2 }numbers.map(n => n * 2) — same higher-order function, same idea, JS just always needs the parens and a named parameter

The "why closures exist" argument from your Swift doc — a function you call gets to decide whether, when, or how many times your code runs, instead of you writing that control flow yourself — carries over to JS completely unchanged. Same idea, same motivation, different syntax:

function retry(times, operation) {
  for (let i = 0; i < times; i++) {
    operation();
  }
}

retry(3, () => console.log("attempt"));
// attempt
// attempt
// attempt

And the "counter factory" example (a closure capturing private state instead of promoting a variable to global scope) works exactly the same way in JS as it does in Swift:

function makeCounter() {
  let current = 0;
  return () => {
    current += 1;
    return current;
  };
}

const counter = makeCounter();
counter(); // 1
counter(); // 2
counter(); // 3

const counter2 = makeCounter(); // independent — its own separate "current"
counter2(); // 1
🔷 Swift comparisonThis is line-for-line the same shape as your makeCounter() -> () -> Int example: current is captured by reference, lives as long as the returned closure lives, and each call to makeCounter() creates an independent capture. JS just doesn't need @escaping to allow this — there's no non-escaping default to opt out of.

The one genuine gap: Swift's capture list lets you snapshot a value at closure-creation time ({ [y] in ... }). JS has no syntax for that — if you want a snapshot instead of a live reference, you make a new variable and let each closure capture its own copy:

// Live reference (JS default — same as Swift's default)
let x = 10;
const printLive = () => console.log(x);
x = 20;
printLive(); // 20 — sees the mutation, same as Swift's default capture

// "Snapshot" — capture a separate variable instead
function snapshot(value) {
  return () => console.log(value); // "value" is its own binding, frozen at call time
}
const printSnapshot = snapshot(x);
x = 30;
printSnapshot(); // still 20 — the parameter copy never changed
⚠️ Watch outThis is the JS equivalent of the classic "loop variable captured by every closure" bug. Fortunately, let/const in a for loop already create a fresh binding per iteration in modern JS (unlike old var), so the naive version usually does what you'd want without extra work — but it's worth knowing why, given how much attention your Swift doc paid to capture semantics.

Destructuring, spread, and rest

These are three separate features that happen to share similar-looking { }, [ ], and ... syntax, which is why they're easy to blur together. Taken one at a time, each is simple:

  • Destructuring — pulling values out of an object or array into their own named variables, in one line instead of one line per field.
  • Spread (...) — taking an existing object or array and expanding its contents into a new object, array, or argument list.
  • Rest (...) — the reverse of spread: gathering up several individual values into a single array or object.

Spread and rest reuse the same ... symbol because they're mirror images of each other — one expands, one collects. Which one you're looking at depends on where the ... appears.

Destructuring objects

Without destructuring, pulling fields off an object means one assignment per field:

const person = { name: "Ada", age: 34 };

// The long way — one line per field
const name = person.name;
const age = person.age;

Destructuring does the same thing in one line. The { name, age } on the left is a pattern that says "look inside whatever's on the right, and create variables matching these property names":

const { name, age } = person;
// name is now "Ada", age is now 34 — identical result to the two lines above

You can rename while destructuring, and give a default if the property might be missing:

const { name: fullName, nickname = "N/A" } = person;
// fullName = "Ada" (renamed from "name")
// nickname = "N/A" (person has no "nickname" property, so the default was used)
🔷 C#/Java comparisonThis is closest to C#'s pattern matching / deconstruction (var (name, age) = person; on a record with a Deconstruct method), except JS destructuring works on plain objects directly, with no Deconstruct method or matching constructor required.

Destructuring arrays

Same idea, but position-based instead of name-based, using [ ] instead of { }:

const scores = [95, 82, 71];

// The long way
const first = scores[0];
const second = scores[1];

// Destructured — identical result
const [first2, second2] = scores;
// first2 = 95, second2 = 82 (71 is simply not captured by anything)

A common real-world use: a function that returns two values as a small array, immediately unpacked into two named variables:

function minMax(nums) {
  return [Math.min(...nums), Math.max(...nums)];
}
const [lowest, highest] = minMax([4, 1, 9, 3]);
// lowest = 1, highest = 9
🔷 C#/Java comparisonThis covers the same use case as C# tuple deconstruction: var (lowest, highest) = MinMax(nums);. JS has no built-in tuple type — it's just destructuring an ordinary two-element array.

Spread — expanding a collection out

Spread takes an existing array or object and unpacks ("spreads") its contents into a new one. The clearest way to see it is combining two arrays without spread first, then with it:

const arrA = [1, 2];
const arrB = [3, 4];

// Without spread — arrA and arrB become nested arrays inside the result, not merged
const wrong = [arrA, arrB];        // [[1, 2], [3, 4]]  — probably not what you wanted

// With spread — each array's elements are unpacked individually into the new array
const combined = [...arrA, ...arrB];  // [1, 2, 3, 4]

Object spread works the same way, and is commonly used to make a modified copy of an object without mutating the original — useful because, as covered earlier, objects are normally mutated in place:

const user = { name: "Ada", age: 34 };

// Copy every property from "user" into a new object, then override just "age"
const updated = { ...user, age: 35 };
// updated = { name: "Ada", age: 35 } — user itself is untouched

Spread also works when calling a function, to unpack an array into separate arguments:

function add3(a, b, c) { return a + b + c; }
const nums = [1, 2, 3];
add3(...nums);   // same as calling add3(1, 2, 3)
🔷 C#/Java comparisonObject spread ({ ...user, age: 35 }) is roughly C#'s with expression on a record (user with { Age = 35 }) — a shallow, non-destructive copy with specific fields overridden. Array/argument spread is similar to C#'s params array being filled by someArray passed with a spread-like unpack, or Java's varargs.
⚠️ Watch outObject/array spread only copies one level deep ("shallow copy"). If a property's value is itself an object or array, both the original and the copy still point at that same nested object — mutating it affects both. Use structuredClone() (from Part 1's array/object methods) when you need a true deep copy.

Rest — gathering values into a collection

Rest is spread's mirror image: instead of unpacking a collection, it gathers multiple individual values into one array. You've actually already seen it, at the end of the array destructuring example — ...rest scoops up everything not already captured by name:

const scores = [95, 82, 71, 60];
const [first, second, ...rest] = scores;
// first = 95, second = 82, rest = [71, 60] — "rest" is a real array containing everything left over

The other place you'll see it is a function parameter list, where it lets a function accept any number of arguments and receive them as one array — this replaces the old, awkward arguments object from pre-2015 JS:

function sum(...nums) {
  // however many arguments are passed, "nums" is a normal array containing all of them
  return nums.reduce((total, n) => total + n, 0);
}
sum(1, 2, 3);        // nums = [1, 2, 3] inside the function → returns 6
sum(10, 20, 30, 40); // nums = [10, 20, 30, 40] → returns 100
🔷 C#/Java comparisonRest parameters (...nums) are directly equivalent to C#'s params int[] nums or Java's varargs int... nums — same purpose, same "collect the remaining arguments into an array" behavior.
✅ Telling them apart quicklySame ... symbol, two directions: on the right-hand side of an assignment or inside a function call, ... is spread (expanding a collection outward). On the left-hand side of an assignment or inside a function's parameter list, ... is rest (gathering values inward). Destructuring is the separate, non-... feature of the { }/[ ] pattern on the left-hand side pulling named or positional values out.

Template literals

const greeting = `Hello, ${name}! You are ${age} years old.`;

// Multi-line strings, no concatenation needed
const html = `
  <div>
    <p>${greeting}</p>
  </div>
`;

Equivalent to C#'s interpolated strings ($"Hello, {name}").

Optional chaining ?. and nullish coalescing ??

const city = user?.address?.city;         // undefined if user or address is null/undefined
const timeout = config.timeout ?? 5000;    // 5000 only if timeout is null or undefined
user?.notify?.();                          // safely call a method that might not exist
⚠️ Watch out?? is not the same as ||. 0 || 5000 gives 5000 (because 0 is falsy). 0 ?? 5000 gives 0 (because 0 is not null/undefined). Prefer ?? for defaults where 0 or "" are valid values.

Classes, private fields, getters/setters

Real class syntax has existed since ES2015, but true private fields (#field) are newer and give you actual encapsulation, not just a naming convention.

class BankAccount {
  #balance = 0;                 // truly private — inaccessible outside the class

  constructor(owner) {
    this.owner = owner;
  }

  get balance() {
    return this.#balance;
  }

  deposit(amount) {
    if (amount <= 0) throw new Error("Amount must be positive");
    this.#balance += amount;
  }

  static openWith(owner, initial) {
    const acct = new BankAccount(owner);
    acct.deposit(initial);
    return acct;
  }
}
🔷 C#/Java comparison#field is closer to true private than JS's old convention of prefixing with _name (which was always just public with a polite request not to touch it). static works the same way you'd expect.

Array & object methods you'll use constantly

const nums = [1, 2, 3, 4, 5];

nums.map(n => n * 2);              // [2,4,6,8,10]
nums.filter(n => n % 2 === 0);     // [2,4]
nums.reduce((acc, n) => acc + n, 0); // 15
nums.find(n => n > 3);             // 4
nums.some(n => n > 4);             // true
nums.every(n => n > 0);            // true
nums.includes(3);                  // true
nums.at(-1);                       // 5 — negative indexing, finally

Object.entries({a: 1, b: 2});      // [["a",1],["b",2]]
Object.keys(obj); Object.values(obj);
Object.fromEntries([["a",1]]);     // {a: 1}

structuredClone(obj);              // real deep clone, built in (no more JSON.parse(JSON.stringify(...)) hacks)
🔷 C#/Java comparisonmap/filter/reduce are LINQ's Select/Where/Aggregate (or Java Streams' map/filter/reduce), except they run eagerly on arrays rather than lazily on an enumerable/stream.

Promises & async/await

This is the single biggest change from 10-years-ago JS. Callback-hell code is gone; async code now reads top-to-bottom.

async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) throw new Error(`HTTP ${response.status}`);
  return response.json();
}

// Calling it
try {
  const user = await getUser(42);
  console.log(user.name);
} catch (err) {
  console.error("Failed to load user:", err);
}

// Running things concurrently
const [users, posts] = await Promise.all([getUsers(), getPosts()]);
🔷 C#/Java comparisonasync/await is directly analogous to C#'s async/await over Task<T> — same mental model, JS Promises ≈ .NET Tasks. Java devs: think CompletableFuture, but with much friendlier syntax.
✅ TipTop-level await now works directly in ES modules — you no longer need to wrap your entry point in an async IIFE just to use await at the top of a file.

ES Modules (import/export)

// math.js
export function add(a, b) { return a + b; }
export const PI = 3.14159;
export default class Calculator { /* ... */ }

// app.js
import Calculator, { add, PI } from "./math.js";

This replaces the old CommonJS require()/module.exports style you may remember from early Node. You'll meet both — see CommonJS vs ESM in Node in Part 2.

Part 2 — Node.js Basics new territory

You said you've never used Node — this part assumes nothing.

What Node actually is

Node.js is a runtime that lets JavaScript run outside a browser — on your machine, on a server, anywhere. It's built on Chrome's V8 engine plus a set of built-in APIs for things browsers don't need (file system access, networking, OS info, etc.).

🔷 C#/Java comparisonNode is to JavaScript roughly what the CLR is to C#, or the JVM is to Java: a runtime + standard library that executes your code and gives you OS-level capabilities. npm-installed packages sit in the role NuGet packages or Maven/Gradle dependencies play.

You already installed it via WebStorm, so you're set up. WebStorm will detect your Node installation automatically (Part 4 covers the exact UI).

npm & package.json

npm (Node Package Manager) ships with Node. Every project has a package.json — its project manifest: name, version, dependencies, and scripts.

{
  "name": "my-app",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "start": "node src/index.js",
    "build": "tsc"
  },
  "dependencies": {
    "express": "^5.0.0"
  },
  "devDependencies": {
    "typescript": "^7.0.2"
  }
}
  • dependencies — packages your app needs at runtime.
  • devDependencies — tools needed only while developing (TypeScript, test runners, linters).
  • node_modules/ — where installed packages actually live. It's generated, huge, and always .gitignore'd — never hand-edit it.
  • package-lock.json — pins exact installed versions for reproducible installs. Commit this to source control.
🔷 C#/Java comparisonpackage.json ≈ your .csproj or Maven pom.xml. node_modules ≈ the NuGet packages cache or your Maven local repo, but scoped per-project rather than shared globally by default.

You said you want to avoid the command line — good news: WebStorm has a full UI for all of this. See Node.js & npm tool windows in Part 4.

Running code without touching a terminal

Covered fully in Part 4, but the short version: you'll right-click a file → Run, or use a Run/Debug configuration and click the green ▶ button in the toolbar. WebStorm generates and manages these for you; you'll rarely need to type a command.

Part 3 — TypeScript complete beginner

Why TypeScript

TypeScript is JavaScript plus a static type system, developed by Microsoft. It compiles ("transpiles") down to plain JavaScript — browsers and Node never run TypeScript directly. Coming from C#/Java, this will feel like coming home: real types, real autocomplete, real compile-time error catching, refactoring you can trust.

✅ TipBecause types are erased at compile time, TypeScript can't catch everything C#/Java's runtime type system catches (e.g., it can't stop you from casting incorrectly at a JSON boundary). Treat TS types as a very good linter, not a runtime guarantee.

Structural typing — the biggest mental shift

C# and Java use nominal typing: a class is only a Shape if it explicitly says implements Shape. TypeScript uses structural typing: if it has the right shape, it counts — no explicit declaration needed.

interface Point {
  x: number;
  y: number;
}

function printPoint(p: Point) {
  console.log(`(${p.x}, ${p.y})`);
}

// No "implements Point" anywhere — this just works,
// because the object has the right shape.
printPoint({ x: 1, y: 2 });

const p3d = { x: 1, y: 2, z: 3 };
printPoint(p3d); // also fine — extra properties are OK when passed via a variable
🔷 C#/Java comparisonThis is the single biggest adjustment. There's no equivalent to structural typing in C#/Java's type systems (the closest is duck typing in dynamic languages, but here it's statically checked). Stop looking for "implements" — look at shape instead.

Basic types & inference

let age: number = 34;
let name: string = "Ada";
let active: boolean = true;
let tags: string[] = ["a", "b"];
let coords: [number, number] = [1, 2];    // tuple — fixed length, fixed types per slot
let anything: unknown = fetchSomething(); // safer than "any" — must narrow before use

// You rarely need to annotate — TypeScript infers from the initializer
let count = 0;          // inferred as number
function double(n: number) {
  return n * 2;          // return type inferred as number
}
⚠️ Watch outAvoid the any type. It silently turns off type checking for that value — you lose everything TypeScript gives you. Use unknown when you genuinely don't know the type yet; it forces you to narrow/check before using it.

Interfaces vs type aliases

interface User {
  id: number;
  name: string;
  email?: string;        // optional property
  readonly createdAt: Date;
}

type UserId = number;                     // alias for a primitive
type Status = "active" | "inactive";      // alias for a union
type Coordinates = { x: number; y: number };

Both interface and type can describe object shapes. Rules of thumb: use interface for objects/classes that might be extended later (interfaces can be reopened and merged); use type for unions, tuples, and anything that isn't a plain extendable object shape.

🔷 C#/Java comparisonDon't map TS interface onto a C# interface one-to-one — a TS interface can describe plain data with no methods at all, more like a lightweight DTO/record shape than a contract a class must implement.

Union types, intersections, and narrowing

type Result = { ok: true; data: string } | { ok: false; error: string };

function handle(result: Result) {
  if (result.ok) {
    console.log(result.data);   // TS knows this branch has "data"
  } else {
    console.log(result.error);  // TS knows this branch has "error"
  }
}

type Named = { name: string };
type Aged = { age: number };
type Person = Named & Aged;      // intersection — must have both

This pattern — a union discriminated by a common literal field (ok above) — is called a discriminated union and is one of TypeScript's most useful idioms. It replaces a lot of what you'd do with class hierarchies or switch on an enum in C#/Java.

Generics

Directly analogous to C#/Java generics — same purpose, similar syntax.

function firstOf<T>(items: T[]): T | undefined {
  return items[0];
}

interface Box<T> {
  value: T;
}

class Stack<T> {
  private items: T[] = [];
  push(item: T) { this.items.push(item); }
  pop(): T | undefined { return this.items.pop(); }
}

const numberStack = new Stack<number>();
🔷 C#/Java comparisonNearly identical mental model to List<T> / generic classes in C# or Java. The main difference: TS generics are erased at compile time (like Java's erasure, unlike C#'s reified generics) — there's no generic type info at runtime at all.

Enums — and why many teams avoid them

enum Direction {
  Up,
  Down,
  Left,
  Right,
}
let d: Direction = Direction.Up;

// Many teams prefer a union of string literals instead:
type Direction2 = "up" | "down" | "left" | "right";
⚠️ Watch outTypeScript's enum generates real runtime JS code (unlike interfaces/types, which vanish entirely at compile time) and has quirky edge cases with numeric enums. The literal-union style above is more idiomatic in modern TS codebases and has zero runtime cost. Know both — you'll see enum in older code.

Classes & access modifiers

class Employee {
  private salary: number;
  protected department: string;
  readonly id: number;

  constructor(id: number, department: string, salary: number) {
    this.id = id;
    this.department = department;
    this.salary = salary;
  }

  // Shorthand: parameter properties do the same thing in one line
  // constructor(public id: number, protected department: string, private salary: number) {}
}
🔷 C#/Java comparisonprivate/protected/public here are compile-time only — they're erased in the emitted JavaScript, so at runtime nothing actually stops access. For true runtime privacy, use JS's native #field syntax from Part 1 instead.

tsconfig.json essentials

This file configures the TypeScript compiler — think of it like project-level compiler settings in a .csproj. WebStorm can generate one for you via the UI (Part 4).

{
  "compilerOptions": {
    "target": "ES2022",        // JS version to compile down to
    "module": "ESNext",        // module system to emit
    "strict": true,             // turn on all strict type-checking — always do this
    "esModuleInterop": true,
    "skipLibCheck": true,
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src/**/*"]
}
✅ TipAlways set "strict": true. It's the difference between TypeScript actually catching bugs and TypeScript being decorative. Everything in this doc assumes strict mode.

TypeScript 7 & the native compiler

As of July 2026, TypeScript 7.0 is stable. The headline change: the compiler itself was rewritten from TypeScript-on-JavaScript to native Go (project codename "Corsa"), giving 8–12× faster full builds and a much more responsive language server on large codebases. The type-checking rules themselves are unchanged from TS 6 — this is a performance rewrite, not a language redesign.

⚠️ Watch outAt TS 7.0's launch, a few framework integrations (Vue, Svelte, Astro, MDX) were still waiting on full editor support pending a missing programmatic API, expected in TS 7.1. Not relevant for plain TS/Node work, but worth knowing if you branch into a frontend framework.

WebStorm 2026.1 can use this native engine via a package called @typescript/native-preview — see TypeScript service settings in Part 4 for how to enable it.

1

3

Running an npm script (e.g., "build": "tsc" from package.json):

1

2

Run/Debug configurations

This is WebStorm's equivalent of Visual Studio's launch profiles.

1

2

3
Point it at your entry file (e.g., src/index.js) via the file picker, give it a name, click OK.
4

Quicker path: right-click any .js/.ts file with a runnable entry point in the editor gutter or Project tree → Run 'filename'. WebStorm creates the configuration automatically.

TypeScript service settings

WebStorm 2026.1 defaults to a service-powered TypeScript engine using the official TypeScript Language Service, which gives faster navigation and more responsive refactoring than older versions. To check or change which engine/version is active:

1

2

To generate tsconfig.json without hand-writing it: right-click the project root → New → File, name it tsconfig.json; WebStorm recognizes the filename and gives you autocomplete + inline documentation for every option as you type.
