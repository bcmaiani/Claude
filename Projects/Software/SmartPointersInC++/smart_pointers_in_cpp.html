Smart Pointers in C++

A living guide to RAII, unique_ptr, shared_ptr, and weak_ptr — starting from the raw-pointer pitfalls that made smart pointers necessary, then building up full, working ownership patterns.

Last updated: 2026-06-30  ·  All examples compiled and run with g++ 11, -std=c++17

Introduction

A smart pointer is a class template that wraps a raw pointer and takes over responsibility for deleting it. Instead of writing new and remembering to pair it with a matching delete everywhere control flow could exit — including exceptions, early returns, and break/continue — you let the smart pointer's destructor do it automatically, exactly once, exactly when the owning variable goes out of scope.

C++11 introduced three smart pointer types in <memory>, and this doc covers all three in depth:

  • std::unique_ptr<T> — sole, non-shareable ownership. The default choice.
  • std::shared_ptr<T> — shared ownership via reference counting. Use when multiple owners genuinely need to co-own an object.
  • std::weak_ptr<T> — a non-owning observer of a shared_ptr-managed object, used to break reference cycles.
Why start with raw pointer pitfalls? Smart pointers aren't a stylistic preference — they exist to close specific, well-documented holes in raw pointer usage. Seeing the failure modes first makes it obvious what problem each smart pointer type is solving, rather than presenting them as syntax to memorize.

The Pitfalls of Raw Pointers

Every one of the problems below compiles cleanly. That's what makes them dangerous — the compiler gives you no warning that anything is wrong, and the failure often shows up far away from its cause, intermittently, or only under load in production.

1. Memory leaks

new must be paired with exactly one delete. If any code path between them exits early — a thrown exception, an early return, a forgotten branch — the delete never runs and the memory is gone for the life of the process.

Raw pointer — leaks when an exception is thrown
#include <iostream>
#include <stdexcept>
#include <string>

struct Widget {
    std::string name;
    Widget(std::string n) : name(std::move(n)) { std::cout << "created " << name << "\n"; }
    ~Widget() { std::cout << "destroyed " << name << "\n"; }
};

void processOrThrow(bool fail) {
    Widget* w = new Widget("Leaky");   // raw pointer, manual ownership
    if (fail) {
        throw std::runtime_error("something went wrong");   // w is never deleted -- LEAK
    }
    delete w;
}

int main() {
    try {
        processOrThrow(true);
    } catch (const std::exception& e) {
        std::cout << "caught: " << e.what() << "\n";
    }
}
// Output:
// created Leaky
// caught: something went wrong
// (no "destroyed Leaky" -- the Widget leaked)
Verified behavior This example was compiled and run exactly as shown. Notice there is no "destroyed Leaky" line — the exception unwinds the stack past the delete w; line, and that memory is never reclaimed for the rest of the program's life.

2. Dangling pointers & use-after-free

A dangling pointer still holds an address, but the memory at that address has already been freed (or the object it pointed to has already been destroyed). Dereferencing it is undefined behavior — it might crash, might silently return garbage, or might "work" until the day it doesn't.

Two pointers to the same heap block, one freed
int* p = new int(10);
int* q = p;        // q points to the same memory as p
delete p;           // memory is freed
p = nullptr;         // p is now safely null...
std::cout << *q;    // ...but q still points to freed memory -- undefined behavior
Why this is so dangerous Setting p = nullptr feels responsible, but it does nothing for q. Nothing in the type system tracks how many raw pointers alias the same block, so nothing can warn you that q is now dangling. This class of bug is a leading source of exploitable security vulnerabilities (use-after-free).

3. Double delete

Calling delete twice on the same pointer is undefined behavior — typically heap corruption, and often a crash that shows up nowhere near the actual bug.

Double delete — undefined behavior
int* p = new int(5);
delete p;
delete p;   // UB: double delete, often corrupts the heap allocator's internal state

4. Unclear ownership

A raw pointer carries no information about who is responsible for freeing it. A function signature like Widget* consume(Widget* w) can't tell the caller: does this function take ownership and delete w? Does it store w somewhere for later? Or is it just borrowing w to read from? You have to go read the implementation — or the documentation, if you're lucky — every single time.

Ambiguous ownership from the signature alone
Widget* create() {
    return new Widget("Mystery");   // caller now owns this... but nothing says so
}

void consume(Widget* w) {
    // Does this function delete w? Store it? Just look at it? The signature can't tell you.
}

5. Exception safety

This is really a special case of the memory leak problem, but it deserves its own callout because it's the one raw-pointer code almost always gets wrong: any function that allocates with new and does anything else afterward — call another function, do arithmetic, touch a container — has to correctly delete on every single exit path, including every possible exception. In non-trivial functions this becomes essentially impossible to audit by hand.

The common thread All five pitfalls above share one root cause: raw pointers require a human to manually keep allocation and deallocation in sync, everywhere, forever, including through control-flow paths that are easy to overlook. Smart pointers move that responsibility into the type system and the compiler.

RAII & The Smart Pointer Idea

A smart pointer is a class that holds a raw pointer internally and calls delete in its own destructor. Because C++ guarantees that destructors run automatically — and deterministically — when a variable goes out of scope, cleanup becomes unconditional. It runs on the normal path, on an early return, and during exception unwinding. This pattern is called RAII: Resource Acquisition Is Initialization — you acquire the resource in a constructor and release it in the matching destructor, so the resource's lifetime is tied to a variable's scope instead of to manual bookkeeping.

The most visible syntactic consequence: the * disappears from the variable declaration and moves into the type name instead.

Raw pointer vs. smart pointer declaration
Widget*                  w = ...;   // raw pointer    -- * in the declaration
std::unique_ptr<Widget> w = ...;   // smart pointer  -- * hidden inside the type name

std::unique_ptr<Widget> still behaves like a pointer — it overloads * and -> so you write *w and w->name exactly as you would with a raw pointer. The difference is entirely in lifetime management, not in day-to-day syntax.

The fixed version of the leak example above
#include <iostream>
#include <stdexcept>
#include <string>
#include <memory>

struct Widget {
    std::string name;
    Widget(std::string n) : name(std::move(n)) { std::cout << "created " << name << "\n"; }
    ~Widget() { std::cout << "destroyed " << name << "\n"; }
};

void processOrThrow(bool fail) {
    std::unique_ptr<Widget> w = std::make_unique<Widget>("Safe");
    if (fail) {
        throw std::runtime_error("something went wrong");
        // w's destructor still runs during stack unwinding -- no leak
    }
}

int main() {
    try {
        processOrThrow(true);
    } catch (const std::exception& e) {
        std::cout << "caught: " << e.what() << "\n";
    }
}
// Output:
// created Safe
// destroyed Safe
// caught: something went wrong
Verified behavior Notice "destroyed Safe" prints before "caught: ...". Stack unwinding runs w's destructor as the exception propagates outward, and only once unwinding reaches the catch block does that block's own code execute. Cleanup is guaranteed regardless of which line throws.

unique_ptr: Basics & make_unique C++11 / C++14

std::unique_ptr<T> owns its object exclusively: exactly one unique_ptr variable owns a given object at any time. It has zero runtime overhead over a raw pointer with the default deleter — no reference count, no extra allocation — which is why it should be the default smart pointer for any single-owner heap allocation.

unique_ptr and make_unique are two distinct things

Both come from #include <memory>, but they play different roles:

  • std::unique_ptr<Widget> is the type — the class that owns the raw pointer and deletes it on destruction.
  • std::make_unique<Widget>("Alpha") is a factory function (C++14) — it calls new Widget("Alpha") for you and hands the result to a unique_ptr, returning that unique_ptr.
Full declaration vs. auto — identical result
// Full declaration -- type is explicit:
std::unique_ptr<Widget> w = std::make_unique<Widget>("Alpha");
// ^^^^^^^^^^^^^^^^^^^^^^      ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
// TYPE of w (owns the ptr)    FACTORY that creates Widget & returns a unique_ptr

// Same thing with auto -- type is deduced, result is identical:
auto w = std::make_unique<Widget>("Alpha");
// auto sees make_unique returns unique_ptr<Widget>, so that becomes the type of w
Always prefer make_unique over unique_ptr<T>(new T(...)) make_unique is exception-safe in ways that a bare new passed to a constructor is not in the presence of certain evaluation-order edge cases involving multiple arguments, and it avoids typing T twice. There is essentially never a reason to write new directly when a factory function is available.

The full lifecycle, annotated

Verified: construction, use, and automatic destruction
#include <memory>
#include <iostream>

struct Widget {
    std::string name;
    Widget(std::string n) : name(n) { std::cout << "created "   << name << '\n'; }
    ~Widget()                        { std::cout << "destroyed " << name << '\n'; }
};

void process(const Widget& w) {   // receives a plain reference, not a pointer
    std::cout << "using " << w.name << '\n';
}

int main() {
    std::unique_ptr<Widget> w = std::make_unique<Widget>("Alpha");
    // Output: created Alpha

    process(*w);   // *w dereferences to a Widget& -- no pointer crosses the call boundary
    // Output: using Alpha

    // main() ends, w goes out of scope: unique_ptr's destructor calls delete on the Widget*
    // Output: destroyed Alpha
}

unique_ptr: Move-Only Ownership

"Exclusive ownership" isn't just documentation — it's enforced by the type. unique_ptr's copy constructor and copy assignment operator are explicitly = deleted, so the compiler rejects any attempt to copy one. It can only be moved, which transfers ownership from the source to the destination and leaves the source null.

Move transfers ownership; the source becomes null
auto a = std::make_unique<int>(1);
auto b = std::move(a);            // ownership transfers to b
std::cout << "a is now " << (a ? "valid" : "null") << "\n";
std::cout << "b holds " << *b << "\n";
// Output:
// a is now null
// b holds 1
Attempting to copy instead -- actual compiler error
auto a = std::make_unique<int>(1);
auto c = a;   // ERROR: unique_ptr copy constructor is deleted
error: use of deleted function
  'std::unique_ptr<_Tp, _Dp>::unique_ptr(const std::unique_ptr<_Tp, _Dp>&)'
note: declared here
  unique_ptr(const unique_ptr&) = delete;
This is a feature, not a limitation The compiler error is the point. It's impossible to accidentally end up with two unique_ptrs that both think they own the same object — the exact bug category from the "unclear ownership" pitfall above is closed off at compile time.

unique_ptr: Custom Deleters

unique_ptr's second template parameter is the deleter type, defaulting to std::default_delete<T> (which just calls delete). You can supply your own deleter to manage any resource that has an "acquire" and "release" function pair — not just heap memory. A common case is a C API resource like a FILE*.

Verified: unique_ptr managing a FILE* with a lambda deleter
#include <cstdio>
#include <memory>
#include <iostream>

int main() {
    auto closer = [](FILE* f) {
        if (f) { std::fclose(f); std::cout << "file closed\n"; }
    };
    std::unique_ptr<FILE, decltype(closer)> file(std::fopen("scratch.txt", "w"), closer);
    if (file) {
        std::fputs("hello\n", file.get());
        std::cout << "wrote to file\n";
    }
    // file closed automatically when it goes out of scope
}
// Output:
// wrote to file
// file closed
Size cost of custom deleters With the default deleter, sizeof(unique_ptr<T>) == sizeof(T*) — zero overhead. A stateless lambda deleter (like the one above) is typically optimized away too via empty base optimization. A stateful deleter (one that captures data, or a raw function pointer) makes unique_ptr grow by however much storage that deleter needs.

unique_ptr: Arrays

unique_ptr has a partial specialization for array types, unique_ptr<T[]>. It calls delete[] instead of delete, and it defines operator[] instead of operator*/operator-> — so it behaves like an array, not a single object.

Verified: unique_ptr<T[]> with make_unique
#include <memory>
#include <iostream>

int main() {
    std::unique_ptr<int[]> arr = std::make_unique<int[]>(5);   // 5 zero-initialized ints
    for (int i = 0; i < 5; ++i) arr[i] = i * i;
    for (int i = 0; i < 5; ++i) std::cout << arr[i] << " ";
}
// Output: 0 1 4 9 16
Never mismatch new/new[] with the wrong unique_ptr form std::unique_ptr<int> p(new int[5]); compiles, but the default deleter calls plain delete on an array allocated with new[] — undefined behavior. Always match unique_ptr<T[]> with array allocation, and prefer std::vector or std::array over raw arrays entirely when you don't have a specific reason to reach for unique_ptr<T[]>.

unique_ptr: Passing & Returning

How you pass a unique_ptr to or from a function communicates ownership intent directly in the signature — this is the fix for the "unclear ownership" pitfall.

SignatureMeaning
void f(Widget* w)Borrows — does not take ownership, may be null
void f(const Widget& w)Borrows — does not take ownership, never null
void f(std::unique_ptr<Widget> w)Takes ownership — caller must std::move in
std::unique_ptr<Widget> f()Transfers ownership out to the caller
Ownership transfer is explicit at the call site
void takeOwnership(std::unique_ptr<Widget> w) {
    // this function now owns w; it will be destroyed when this function returns
    // (unless it moves w somewhere else first)
}

std::unique_ptr<Widget> factory() {
    return std::make_unique<Widget>("Built");   // ownership moves out to the caller
}

int main() {
    auto w = factory();          // main() now owns the Widget
    takeOwnership(std::move(w)); // explicit -- you can SEE ownership leaving at the call site
    // using w here would be a compile error opportunity for a linter/reviewer to catch --
    // w is valid but null after the move, so *w would be undefined behavior if dereferenced
}
If a function only needs to look at the object, don't take a unique_ptr at all Prefer const Widget& or Widget* for functions that merely observe. Reserve unique_ptr parameters for functions that are genuinely taking ownership — passing it otherwise forces every caller to std::move, which is a strong (and correct) signal that something permanent just happened.

shared_ptr: Reference Counting C++11

unique_ptr enforces exactly one owner. std::shared_ptr<T> relaxes that: any number of shared_ptr instances can co-own the same object via an atomic reference count. The object is destroyed only when the last owning shared_ptr is destroyed or reset.

Verified: use_count() tracking ownership
#include <memory>
#include <iostream>

int main() {
    auto a = std::make_shared<int>(42);
    std::cout << "after a: use_count = " << a.use_count() << "\n";
    {
        auto b = a;   // b shares ownership -- copy increments the count
        std::cout << "after b = a: use_count = " << a.use_count() << "\n";
        b.reset();    // b releases its share
        std::cout << "after b.reset(): use_count = " << a.use_count() << "\n";
    }
    std::cout << "final use_count = " << a.use_count() << "\n";
}
// Output:
// after a: use_count = 1
// after b = a: use_count = 2
// after b.reset(): use_count = 1
// final use_count = 1
Copying is the whole interface Unlike unique_ptr, shared_ptr's copy constructor is perfectly normal — copying is exactly how you create a second owner. Each copy atomically increments the shared count; each destruction or reset() atomically decrements it.

shared_ptr: The Control Block

Every group of shared_ptrs that co-own an object shares a single heap-allocated control block, which holds the strong reference count, the weak reference count (see weak_ptr below), the deleter, and (for make_shared) the object itself.

make_shared: one allocation instead of two
// Two separate heap allocations: one for Widget, one for the control block
std::shared_ptr<Widget> a(new Widget("Alpha"));

// One combined heap allocation for both the control block AND the Widget
auto b = std::make_shared<Widget>("Alpha");
Prefer make_shared It's faster (one allocation instead of two) and more exception-safe for the same reason make_unique is. The one tradeoff: because the object and control block share a single allocation, the object's memory isn't actually freed until every shared_ptr and every weak_ptr referencing it are gone — a lingering weak_ptr can keep a large object's memory reserved (though not the object itself, which is destroyed once the strong count hits zero).

The aliasing constructor

shared_ptr has a constructor that shares the control block (and thus the reference count) of one shared_ptr while pointing at a different address — typically a member of the owned object. This lets you hand out a shared_ptr to a sub-object while keeping the parent alive for as long as that pointer exists.

Aliasing constructor: point at a member, keep the owner alive
struct Engine { int horsepower = 300; };
struct Car { Engine engine; };

auto car = std::make_shared<Car>();
std::shared_ptr<Engine> enginePtr(car, &car->engine);
// enginePtr.use_count() participates in the same control block as car --
// the Car stays alive as long as either shared_ptr exists, even though
// enginePtr's stored address points at the Engine sub-object, not the Car.

shared_ptr: Custom Deleters

Unlike unique_ptr, the deleter is not part of shared_ptr's type — it's stored (type-erased) in the control block. That means two shared_ptr<T>s with different deleters still have the same type, and you can freely pass either where a shared_ptr<T> is expected.

Custom deleter passed to the constructor
std::shared_ptr<Widget> w(new Widget("Custom"), [](Widget* p) {
    std::cout << "custom cleanup for " << p->name << "\n";
    delete p;
});
make_shared can't take a custom deleter Because make_shared combines the object and control block into one allocation, there's no separate "delete this object" step to override — you always get the default deleter with make_shared. If you need a custom deleter, construct the shared_ptr directly from a raw pointer as shown above (accepting the two-allocation cost).

weak_ptr: Breaking Cycles C++11

shared_ptr solves shared ownership, but it introduces a new failure mode: if two objects hold shared_ptrs to each other, neither's strong count ever reaches zero, and neither is ever destroyed — a memory leak, even though nothing is reachable from outside the cycle anymore.

Verified: two shared_ptrs pointing at each other leak
#include <memory>
#include <iostream>

struct Child;
struct Parent {
    std::shared_ptr<Child> child;
    ~Parent() { std::cout << "Parent destroyed\n"; }
};
struct Child {
    std::shared_ptr<Parent> parent;   // BUG: strong back-reference creates a cycle
    ~Child() { std::cout << "Child destroyed\n"; }
};

int main() {
    auto p = std::make_shared<Parent>();
    auto c = std::make_shared<Child>();
    p->child = c;
    c->parent = p;
    std::cout << "parent use_count=" << p.use_count() << " child use_count=" << c.use_count() << "\n";
    std::cout << "leaving scope...\n";
}
// Output:
// parent use_count=2 child use_count=2
// leaving scope...
// (nothing else -- NEITHER destructor ever runs. Both objects leak.)

The fix: make one direction of the relationship non-owning with std::weak_ptr<T>. A weak_ptr can observe an object managed by a shared_ptr without contributing to the strong count, so it doesn't keep the cycle alive.

Verified: the same structure, fixed with weak_ptr
struct Child;
struct Parent {
    std::shared_ptr<Child> child;
    ~Parent() { std::cout << "Parent destroyed\n"; }
};
struct Child {
    std::weak_ptr<Parent> parent;   // FIX: non-owning back-reference breaks the cycle
    ~Child() { std::cout << "Child destroyed\n"; }
};

int main() {
    auto p = std::make_shared<Parent>();
    auto c = std::make_shared<Child>();
    p->child = c;
    c->parent = p;   // weak_ptr does not increment the strong ref count
    std::cout << "parent use_count=" << p.use_count() << " child use_count=" << c.use_count() << "\n";
    std::cout << "leaving scope...\n";
}
// Output:
// parent use_count=1 child use_count=2
// leaving scope...
// Parent destroyed
// Child destroyed
Rule of thumb for the cycle direction In any parent/child, owner/observer, or tree-with-parent-links structure, let ownership flow in one direction only (e.g. parent → child via shared_ptr) and make the reverse link (child → parent) a weak_ptr.

weak_ptr: Observing Without Owning

Because a weak_ptr doesn't keep its object alive, you can never dereference one directly — there's no operator* or operator->. Instead, call .lock(), which atomically checks whether the object is still alive and, if so, returns a shared_ptr to it (which now safely keeps it alive for as long as you hold that returned shared_ptr). If the object is already gone, .lock() returns an empty shared_ptr.

Verified: lock() after the owning shared_ptr is gone
#include <memory>
#include <iostream>

int main() {
    std::weak_ptr<int> weak;
    {
        auto strong = std::make_shared<int>(99);
        weak = strong;
        if (auto locked = weak.lock()) {
            std::cout << "still alive, value = " << *locked << "\n";
        }
    }   // strong goes out of scope here -- the int is destroyed
    if (auto locked = weak.lock()) {
        std::cout << "still alive: " << *locked << "\n";
    } else {
        std::cout << "expired: object is gone\n";
    }
}
// Output:
// still alive, value = 99
// expired: object is gone
Don't use expired() then dereference separately Checking weak.expired() and then calling .lock() in a second step has a race in multithreaded code — the object could be destroyed between the two calls. Always call .lock() once, store the result, and check that, as shown above.

Other uses beyond cycle-breaking

weak_ptr is also the right tool for caches and observer/listener lists — anywhere you want to reference an object without forcing it to stay alive just because something is watching it. A cache that holds weak_ptrs naturally evicts entries once nothing else references the underlying object.

Casting Smart Pointers

<memory> provides shared_ptr equivalents of the standard C++ cast operators. Each one shares the same control block as the original, so the reference count stays correct across the cast.

FunctionEquivalent to
std::static_pointer_cast<T>static_cast
std::dynamic_pointer_cast<T>dynamic_cast — returns null shared_ptr on failure
std::const_pointer_cast<T>const_cast
std::reinterpret_pointer_cast<T>reinterpret_cast C++17
Verified: dynamic_pointer_cast for a downcast
#include <memory>
#include <iostream>

struct Shape { virtual ~Shape() = default; };
struct Circle : Shape { double radius = 2.5; };
struct Square : Shape { double side = 4.0; };

int main() {
    std::shared_ptr<Shape> s = std::make_shared<Circle>();

    if (auto c = std::dynamic_pointer_cast<Circle>(s)) {
        std::cout << "it's a circle, radius = " << c->radius << "\n";
    }
    if (auto sq = std::dynamic_pointer_cast<Square>(s)) {
        std::cout << "it's a square, side = " << sq->side << "\n";
    } else {
        std::cout << "not a square\n";
    }
}
// Output:
// it's a circle, radius = 2.5
// not a square

get() / release() / reset()

Three member functions cover interop with raw-pointer APIs and manual lifecycle control. They exist on both unique_ptr and shared_ptr (with slightly different implications for shared_ptr, since other owners may still exist).

CallEffect
p.get()Returns the raw pointer without releasing ownership. The smart pointer still owns and will still delete it.
p.release()unique_ptr only. Releases ownership and returns the raw pointer; p becomes null. The caller is now responsible for the memory.
p.reset(newPtr)Destroys the currently owned object (if any) and takes ownership of newPtr (or nothing, if omitted).
Verified: all three in one flow, including a legacy-API handoff
#include <memory>
#include <iostream>

void legacyApiTakesOwnership(int* p) {
    std::cout << "legacy API now owns value " << *p << "\n";
    delete p;   // the legacy API is responsible for freeing it
}

int main() {
    auto up = std::make_unique<int>(7);
    std::cout << "get() peek without releasing: " << *up.get() << "\n";
    std::cout << "up still valid? " << (up ? "yes" : "no") << "\n";

    int* raw = up.release();          // up gives up ownership, becomes null
    std::cout << "up still valid after release()? " << (up ? "yes" : "no") << "\n";
    legacyApiTakesOwnership(raw);      // caller (legacy API) must now delete it

    up.reset(new int(100));            // reset takes ownership of a new object
    std::cout << "after reset(): " << *up << "\n";
}
// Output:
// get() peek without releasing: 7
// up still valid? yes
// up still valid after release()? no
// legacy API now owns value 7
// after reset(): 100
get() is a loan, not a transfer Never call delete on the result of .get() — the smart pointer still thinks it owns that memory and will delete it too, causing a double delete. Only .release() gives up ownership.

enable_shared_from_this

Sometimes a member function needs to hand out a shared_ptr to *this — for example, to register a callback that should keep the object alive. Naively wrapping this in a brand-new shared_ptr creates a second, independent control block that doesn't know about the original one, leading to a double delete once both control blocks separately reach zero.

Inheriting from std::enable_shared_from_this<T> fixes this: it gives the object a hidden weak_ptr back to itself, populated automatically the first time the object is wrapped in a shared_ptr. Calling shared_from_this() then locks that weak_ptr, correctly reusing the existing control block.

Verified: shared_from_this() correctly shares the existing control block
#include <memory>
#include <iostream>

struct Node : std::enable_shared_from_this<Node> {
    std::shared_ptr<Node> getSelf() { return shared_from_this(); }
};

int main() {
    auto n = std::make_shared<Node>();
    std::cout << "use_count before: " << n.use_count() << "\n";
    auto self = n->getSelf();
    std::cout << "use_count after getSelf(): " << n.use_count() << "\n";
    std::cout << (n.get() == self.get() ? "same object\n" : "different object\n");
}
// Output:
// use_count before: 1
// use_count after getSelf(): 2
// same object
Must already be owned by a shared_ptr If you call shared_from_this() on an object that isn't currently owned by any shared_ptr (e.g. a stack-allocated Node n;), it throws std::bad_weak_ptr. This pattern only works for objects created via make_shared or a shared_ptr constructor.

Common Mistakes — With Smart Pointers Themselves

Smart pointers close off the raw-pointer pitfalls from the top of this doc, but they introduce their own sharp edges. These are the ones that bite most often.

1. Constructing two shared_ptrs from the same raw pointer

BUG: two independent control blocks, same object -- double delete
Widget* raw = new Widget();
std::shared_ptr<Widget> a(raw);
std::shared_ptr<Widget> b(raw);   // BUG: b doesn't know about a's control block
// When a and b both go out of scope, EACH deletes raw independently -- crash.
// Fix: std::shared_ptr<Widget> b = a;  (copy, sharing the same control block)

2. Manually wrapping this in a shared_ptr

Same root cause as above. Use enable_shared_from_this instead of constructing std::shared_ptr<Widget>(this) by hand.

3. Reference cycles

Covered in depth above — two shared_ptrs pointing at each other never reach a zero count. Break the cycle with a weak_ptr on one side of the relationship.

4. Storing the result of .get() beyond the smart pointer's lifetime

Widget* dangerous;
{
    auto w = std::make_unique<Widget>("Temp");
    dangerous = w.get();   // raw pointer, valid only while w is alive
}   // w destroyed here -- dangerous now points at freed memory
// *dangerous is undefined behavior from this point on

5. Passing unique_ptr by value when you don't need ownership transfer

Forcing every caller to std::move in just to let your function look at the object is unnecessary friction. If you don't need ownership, take const Widget& or Widget* instead.

6. Mixing new[]/delete[] with the wrong unique_ptr specialization

Covered above — always pair array allocation with unique_ptr<T[]>, never plain unique_ptr<T>.

The unifying fix for all six Every mistake above comes from bypassing the smart pointer's own bookkeeping — reaching for a raw new/this/.get() and treating it as if it were independent of the object the smart pointer already manages. Stay inside the smart pointer API (copy, move, shared_from_this(), weak_ptr) and these can't happen.

When to Use Which

Use unique_ptr when…

  • Exactly one piece of code is ever responsible for the object
  • You want zero runtime overhead over a raw pointer
  • You're returning a freshly allocated object from a factory function
  • You're storing a polymorphic object in a container (std::vector<std::unique_ptr<Base>>)
  • This is your default choice — reach for it first

Use shared_ptr when…

  • Multiple independent owners genuinely need to keep the object alive
  • Object lifetime is determined by "whoever uses it last," not a single scope
  • You're building a graph/cache/plugin system with shared, ref-counted resources
  • You've confirmed unique_ptr doesn't fit — don't default to shared_ptr

Use weak_ptr when…

  • You need a back-reference in a structure that would otherwise cycle
  • You want to observe an object without forcing it to stay alive
  • You're building a cache or listener list of shared_ptr-managed objects

Use a raw pointer (or reference) when…

  • You're just observing — not managing lifetime at all
  • You're working with a C API that expects a plain address
  • Passing the smart pointer itself would add nothing — pass T&/T* instead
One-line summary Default to unique_ptr for ownership. Reach for shared_ptr only when ownership is genuinely shared. Use weak_ptr to observe or to break a cycle. Use a raw pointer or reference for everything that's just borrowing, not owning.

Best Practices Checklist

  • Never write a bare new or delete in application code — use make_unique/make_shared, or a container, instead.
  • Default to unique_ptr; upgrade to shared_ptr only when you've confirmed you need multiple independent owners.
  • Prefer make_unique/make_shared over constructing a smart pointer from a raw new — fewer allocations, better exception safety.
  • Take smart pointers as function parameters only when the function is genuinely taking or sharing ownership; otherwise take T&/T*.
  • Break any potential reference cycle by making one direction of the relationship a weak_ptr.
  • Inherit from enable_shared_from_this instead of manually wrapping this in a shared_ptr.
  • Never call delete on the result of .get() — it doesn't transfer ownership.
  • Match unique_ptr<T[]> with array allocations; prefer std::vector/std::array when there's no specific reason not to.
