Overview

"Duplicate the iOS app in React Native" isn't a file copy — Swift/SwiftUI/SwiftData code doesn't run in a JS engine, so this was a full re-implementation aimed at functional parity: same two tabs, same session → task → recording data model, same timer and analytics behavior, same file-storage approach for recordings. This doc records what was ported, what technology stood in for each Swift/Apple API, and what still needs manual verification on your machine.

The source app is a single-screen-flow music practice tracker: log practice sessions, break each into tasks, time each task with a start/pause timer, attach audio/video recordings to a task, and see 30-day/day-of-week trend charts plus a streak counter in a History tab.

Technology Mapping

Every Apple-specific API/framework in the source app needed a React Native/Expo equivalent. These were the substitutions made, and why:

Swift / AppleReact Native / ExpoWhy
SwiftUIReact Native components + StyleSheetDirect UI-layer equivalent
SwiftData (@Model, @Query, cascade @Relationship) expo-sqlite with hand-written SQL + ON DELETE CASCADE foreign keys No direct RN ORM has SwiftData's ergonomics; raw SQLite with explicit cascade FKs reproduces the same cascade-delete guarantee at the DB layer. A small pub/sub bus (src/db/events.ts) fakes @Query's live-updating reads, since SQLite has no push notifications.
NavigationStack + TabView React Navigation (@react-navigation/native-stack + bottom-tabs) Closest structural match — stacks nested inside tabs, same as the source.
Swift Charts (BarMark)react-native-gifted-charts SVG-only dependency (via react-native-svg), no native linking/Skia required — keeps the project runnable in Expo Go.
AVAudioRecorderexpo-av's Audio.Recording Closest maintained Expo audio-recording API at the time of writing.
UIImagePickerController (camera, video mode) expo-image-picker's launchCameraAsync Direct RN equivalent for camera-sourced video capture.
Application Support directory + isExcludedFromBackup expo-file-system's documentDirectory Closest app-private, persistent storage location. No RN equivalent for excluding a directory from device backups — see Known Limitations.
TimelineView(.periodic(from:by:))useTicker hook (setInterval) Forces a once-per-second re-render for the live timer display.

Data Layer: SwiftData → SQLite

The source app's three @Model classes (PracticeSession, SessionTask, Recording) became three SQLite tables with foreign keys instead of SwiftData relationships:

src/db/database.ts (schema)
CREATE TABLE tasks (
  id TEXT PRIMARY KEY NOT NULL,
  session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  date INTEGER NOT NULL,
  task_description TEXT NOT NULL DEFAULT '',
  time_spent REAL NOT NULL DEFAULT 0,
  timer_started_at INTEGER
);

ON DELETE CASCADE reproduces SwiftData's @Relationship(deleteRule: .cascade) — deleting a session automatically deletes its tasks and their recordings' metadata rows. Just like the Swift version, the DB layer has no idea Recording rows point at files on disk, so sessionsRepository.deleteSession() and tasksRepository.deleteTask() explicitly delete the media files first, mirroring the Swift performDelete() methods' file-cleanup comments.

Since raw SQLite has no live-query push mechanism (unlike @Query), reactivity is faked with a tiny pub/sub bus: repositories call notifyChange(table) after every mutation, and hooks like useSessions() subscribe and re-fetch.

The timer itself ported almost line-for-line: timerStartedAt (epoch ms) + accumulated timeSpent (seconds) is the same wall-clock approach as the Swift Date/TimeInterval pair, so it survives the JS thread being backgrounded exactly like the original survives iOS suspending the app.

Services

src/services/practiceAnalytics.ts is a deliberate line-for-line port of PracticeAnalytics.swift — same four pure functions (dailyTotals, weekdayAverages, currentStreak, suggestion), same streak logic (counts from yesterday if today isn't practiced yet), same "best vs. worst weekday" and "N of last 7 days" suggestion heuristics.

src/services/useAudioRecorder.ts ports AudioRecorder.swift's isRecording/elapsed/lastError state shape and start()/stop()/cancel() actions as a React hook backed by expo-av, polling recording status every 200ms — the same interval the Swift version used for its display-timer poll loop.

Navigation & Screens

Structure mirrors the source 1:1 — a bottom-tab root (RootTabNavigator) with a Practice stack (SessionList → SessionDetail → TaskDetail) and a History stack, matching RootTabView.swift's TabView + nested NavigationStacks. Each SwiftUI view became one screen component; swipe-to-delete uses react-native-gesture-handler's Swipeable, gated behind the same "only confirm if a recording would be destroyed" logic as the Swift .swipeActions + .confirmationDialog combination.

Tests

__tests__/practiceAnalytics.test.ts and __tests__/timer.test.ts port the assertions from PracticeAnalyticsTests.swift and the timer/formatting portion of ModelTests.swift — same fixed reference instant, same expected day-total/streak values.

Not ported: ModelTests.swift's SwiftData cascade-delete test. It requires expo-sqlite's native binding running on-device/in-simulator, not a plain Node/Jest environment. Verify cascade deletes manually (create a session with a recording, delete it, confirm the file and DB rows are both gone) or add a Detox/Maestro end-to-end test.

Verification Performed

No macOS/Xcode toolchain or full Node/Expo install was available in the sandbox this was written in (same caveat the original Swift README carries about not having a Mac toolchain), so verification was scoped to what could actually run:

  • Strict TypeScript check on the pure logic modules (practiceAnalytics.ts, timer.ts, types.ts, events.ts, uuid.ts) — compiled clean with strict: true.
  • Runtime execution of the compiled analytics/timer functions against the same fixed inputs used in the Swift unit tests — outputs matched exactly (daily totals with gaps filled as 0, streak = 3/2/0 for the three streak scenarios, duration formatting for sub-minute/sub-hour/multi-hour durations).
  • File-tree check confirming every planned file exists at the target path.
Not verified: full npm install (the RN/Expo dependency tree is large enough that it didn't complete within this environment's command time limits) and an actual Metro bundle/simulator run. Do npm install on your machine per the Running It in WebStorm section below, then work through whatever errors npx expo install --check and the Metro bundler surface — most likely spots are native-module version mismatches, same class of issue the source README flagged for the Xcode build.

Known Limitations

  • No push-based live queries. The pub/sub-plus-refetch approach in src/db/events.ts is fine at this app's scale but isn't a real reactive database layer — consider WatermelonDB or Drizzle's useLiveQuery if the dataset grows much larger.
  • No backup-exclusion flag on the recordings directory. expo-file-system has no equivalent to URLResourceValues.isExcludedFromBackup; recordings will be included in standard device backups unless a small native module is added for this.
  • Tab icons are emoji, not SF Symbols. Swap in @expo/vector-icons for closer visual parity if desired.
  • No cross-device sync, matching the source app's CloudKit-disabled state — both are local-only for now.
  • Not build-verified end-to-end — see Verification Performed above.

Running It in WebStorm

1Open the project

File → Open → select /Users/bradmaiani/WebstormProjects/MusicPracticeTracker.

2Install the React Native Console plugin

Preferences/Settings → Plugins → Marketplace → search "React Native Console" → Install → restart WebStorm.

3Install dependencies
npm install
4Start Metro
npm start

The React Native Console tool window attaches to this Metro process and streams console.log output plus JS errors/warnings from the running app.

5Run on a simulator/device
npm run ios     # macOS + Xcode simulator
npm run android # Android emulator or device
Recording features need a physical device (or a simulator with camera passthrough) — same constraint the source app's README calls out for its own simulator limitations.
