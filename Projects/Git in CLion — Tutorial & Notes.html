Git in CLion, without the bafflement

The branch → commit → merge → sync loop done entirely in CLion's GUI, plus a field guide to the four traps that turned one comment into a two-hour ordeal. Every trap here is one you actually hit — and none was a failure of yours.

LIVING DOCUMENT · v2 · updated per session

01 The model, fast

Almost all the confusion comes from never being shown the one underlying picture. Git is three things:

A commit
A saved snapshot of your files with a message. Permanent. Each has an ugly ID like cc10eae1. Commits never move.
A branch
Just a movable label that points at a commit. main is a label. When you commit, the label slides forward. That's all a branch is.
HEAD
A pointer marking where you are now. Normally it points at a branch label, and the label points at a commit (HEAD → branch → commit). When it points straight at a commit, skipping the branch label (HEAD → commit), that's detached HEAD, trap #1.
Why this matters Every trap tonight is one question: is HEAD attached to a branch label, or not? The single tell is the widget in CLion's top header — a branch name (attached, fine) or an ID like cc10eae1 with a yellow ⚠ triangle (detached, fix before continuing).

02 The core loop: branch, commit, merge

The branch widget lives in the top window header (main toolbar), not the bottom status bar — that's the modern CLion UI. Clicking it opens the VCS widget (Git Branches popup). Everything below happens there.

A · Make a branch

  1. Click the branch name (e.g. main) in the top header to open the VCS widget.
  2. Under Local, click main, then New Branch from 'main'.
  3. Name it descriptively, e.g. add-feature-x. Leave Checkout branch ticked. Click Create.
  4. Confirm the header now shows your new branch name and no yellow triangle.

B · Make changes & commit

  1. Edit files normally. Changed files turn blue in the Project view.
  2. Open the Commit tool window: ⌘0 (or View ▸ Tool Windows ▸ Commit).
  3. Tick the files to include. Untick workspace.xml and .idea/ files — IDE clutter, not your code (trap #4).
  4. Type a message. Click Commit (not Commit and Push — keep steps separate while learning).

C · Merge the branch into main

  1. Open the VCS widget. Under Local, click main → Checkout. Confirm the header reads main.
  2. Open the VCS widget again. Under Local, click your feature branch.
  3. Choose Merge '…' into 'main'. The wording must say into 'main' — if it says into an ID, stop (detached — trap #1).
  4. Done. Because main hadn't moved, this is instant and conflict-free.
What "done" looks like Header reads main, no triangle. The Git Log (⌥9 ▸ Log) shows your feature commit now on the main line. Nothing pending.

03 Syncing across your two Macs

Your real goal: GitHub as the shared source of truth, both Macs interchangeable. One rhythm prevents the divergence trap:

Start of every session

  1. Git ▸ Pull ⌘T — pull first, before touching anything, so local matches GitHub.
  2. Now branch, commit, merge as in §2.

End of every session

  1. Make sure everything is committed (Commit window empty of your code files).
  2. Git ▸ Push ⇧⌘K — send your work up to GitHub.
Why the order matters Divergence (trap #3) happens when both GitHub and local main have commits the other lacks. Pull-before-work, push-after-work means they never drift apart. Tonight's ordeal came from merging on one machine then pulling on the other without this rhythm — normal for a first attempt, and exactly what the rhythm prevents.
One-time setting CLion ▸ Settings ⌘, ▸ Version Control ▸ Git ▸ set Update method to Merge. The safe default; reduces how often the "divergent branches" prompt appears.

04 Ignoring IDE files (kill trap #4 for good)

Files like workspace.xml are CLion's own settings — window layout, open tabs, recent files — not your project's code. They change every time you use the IDE, which is why they keep blocking merges and, across your two Macs, generate endless phantom conflicts. The fix is to tell Git to stop tracking them via a file called .gitignore.

What a .gitignore is

A plain text file named exactly .gitignore, living in your project's top folder. Each line names a file or folder pattern Git should ignore entirely — never track, never commit, never complain about. JetBrains officially recommends ignoring the IDE files below.

A · Create the file in CLion (GUI)

  1. In the Project tool window (⌘1), right-click your project's top-level folder.
  2. Choose New ▸ File.
  3. Name it exactly .gitignore — with the leading dot, no extension. Press Enter.
  4. If CLion asks whether to add it to Git, click Add (this file should be tracked — it's the one IDE-related file you keep).

B · Paste these lines in

Open the new .gitignore and paste the following. The lines starting with # are comments (ignored by Git, just notes for you):

# ---- JetBrains IDE files (CLion, Rider, IntelliJ) ----
# The whole .idea folder is per-machine settings, not code.
.idea/

# If you ever want to KEEP a couple of shared .idea files instead
# of ignoring the whole folder, delete the line above and use these:
# .idea/workspace.xml
# .idea/tasks.xml
# .idea/usage.statistics.xml
# .idea/shelf/

# ---- macOS junk ----
.DS_Store

# ---- Build output (CMake / CLion) ----
cmake-build-*/
build/
If the file was already committed Ignoring a file doesn't remove one Git is already tracking — like workspace.xml tonight. You also need to untrack it once: in the Commit tool window or Project view, right-click workspace.xml (or the .idea folder) → Git ▸ Rollback / Unversion, or use Git ▸ Add to .gitignore which CLion offers directly on a right-clicked file. After that, commit the change. From then on it's silent.

C · The shortcut CLion gives you

You don't always have to write the file by hand. When an IDE file shows up in your changes, right-click it in the Commit window and look for Git ▸ Add to .gitignore. CLion appends the right line for you and creates the .gitignore if it doesn't exist. Either route gets you to the same place.

D · Commit the .gitignore

  1. Open the Commit window (⌘0). You'll see .gitignore listed.
  2. Tick it, write a message like Add gitignore for IDE files, and Commit.
  3. Git ▸ Push so the other Mac gets the same rules. Now both machines ignore the same clutter.
Result workspace.xml and friends stop appearing in every commit and stop blocking merges. Trap #4 is gone on both Macs — permanently.

05 Field guide to the four traps

The exact four you hit, in the order they bit you. Click to expand.

Trap 1 · Detached HEAD header shows cc10eae1 + ⚠

What you see

Top header shows an ID like cc10eae1 instead of a branch name, with a yellow ⚠. Merge options read "into cc10eae1" not "into 'main'."

What it actually means

HEAD points straight at a commit instead of a branch label. You're standing on a snapshot with no branch under your feet; commits made here aren't attached to a branch.

How you got here

You checked out a commit rather than a branch — right-clicking a commit in the Log and choosing Checkout, selecting a commit-row instead of a branch name, or checking out an origin/… line directly.

The fix (GUI)

First: did you commit anything while detached? Open the Log (⌥9). If a commit sits at the top with no branch label, rescue it: VCS widget → New Branch → name it rescue → Checkout. Then reattach: VCS widget → Local → main → Checkout. Header returns to main, triangle gone. If nothing was committed while detached (your case tonight), checkout main directly — nothing lost.

Trap 2 · The branch that "vanished" can't find my branch

What you see

Your branch isn't where you expect; the Local node doesn't list the name you're hunting.

What it actually means

Almost always a name mismatch, not a deletion. Tonight you searched for new_feature (a placeholder name) while your branch was actually Add-Branching-Comment. It was in the list the whole time under its real name.

The fix (GUI)

Open the Log (⌥9 ▸ Log). Branch names show as colored tags next to commits — read the actual names. The Local node is the authoritative list; trust it over any half-remembered name.

Prevention

Name branches deliberately and note the name for the session. A label only helps if you know what it says.

Trap 3 · Divergent branches "specify how to reconcile"

What you see

A pull fails with "You have divergent branches and need to specify how to reconcile them."

What it actually means

Not an error — a question. Both local and GitHub copies have commits the other lacks, so Git refuses to guess and asks you to choose. Triggered by merging locally on one Mac, then pulling.

The fix (GUI)

Set the default once: Settings ⌘, ▸ Version Control ▸ Git ▸ Update method = Merge. If a pull still balks, split it: Git ▸ Fetch, then VCS widget → Remote ▸ origin ▸ main → Merge 'origin/main' into 'main'. Naming the merge yourself sidesteps the question.

Prevention

The pull-first / push-last rhythm in §3.

Trap 4 · workspace.xml in the way "unsaved changes to workspace.xml"

What you see

A merge or checkout stalls over uncommitted changes to workspace.xml — a file you never touched.

What it actually means

workspace.xml (inside .idea/) is a CLion settings file — window layout, open tabs, recent files. Not your code. It changes constantly from just using the IDE, so it keeps blocking operations.

The fix right now (GUI)

When prompted, choose Smart checkout / Shelve / Stash — any sets it aside, runs your operation, restores it. Safe; only IDE layout moves, never code.

The permanent fix

Stop tracking it: add .idea/workspace.xml to .gitignore (JetBrains recommends ignoring it, especially across two machines where it causes endless spurious conflicts). Save for a calm moment — see session notes.

06 Detached HEAD: causes & escape

This is the one that has caught you more than once, so here it is in full — every specific action that causes it, and the exact steps out. Nothing here requires the terminal.

The one underlying cause

HEAD detaches for exactly one reason: you checked out something that is not a local branch. A local branch is a movable label (like main). A commit, a tag, and a remote ref like origin/main are not movable branch labels — landing on any of them points HEAD straight at a commit with no branch, which is the detached state. There is no other way to detach. So the whole skill is: only ever check out a branch name from the Local node.

The specific actions that detach you (avoid these)

Checkout from the Log

Right-clicking a commit in the Git Log (⌥9) and choosing Checkout Revision. The most common cause. You go to the Log to look at an old state, and Checkout feels like the way — but it lands you on that commit, detached.

Checkout Tag or Revision…

Using the VCS widget's Checkout Tag or Revision… entry and giving it a commit ID or a tag. Tags always detach — a tag is a fixed marker, not a branch.

Selecting a commit row

Clicking a commit line instead of the branch name in a Git panel, then confirming a checkout. The commit and the branch label sit close together visually — an easy misclick.

Checking out origin/… directly

Checking out a remote ref like origin/main from the Remote node. These are read-only pointers; landing on one directly can detach you. If CLion offers to make a matching local branch, accept it.

The single habit that prevents all of them

  1. Open the VCS widget (branch name in the top header).
  2. Go to the Local node — not Remote, not the Log.
  3. Click the branch name you want (e.g. main, add-feature-x).
  4. Choose Checkout.
  5. Glance at the header: it should show that branch name, no ⚠. If it shows an ID, you detached — see escape below.
If you only want to LOOK at an old commit That's the usual reason people go to the Log and get burned. You don't need to check it out to look. Just click the commit in the Log — its diff and files show in the side panel, no checkout, no detaching. And if you genuinely need to work from an old commit, right-click it → New Branch… (not Checkout). That puts a branch label on it and lands you on the branch — attached and safe.

How to get out of it — the escape

The instant the header shows an ID like cc10eae1 with a ⚠, do this. The order matters, because step 1 protects any work before you move.

  1. Check whether you committed anything while detached. Open the Log (⌥9 ▸ Log). Look at the very top commit. Does it have a branch-name tag on it, or is it tagless and sitting above where main points?
  2. If there's a tagless commit at the top (work you'd lose): rescue it first. Open the VCS widget → New Branch → name it rescue → keep Checkout ticked → Create. Your work is now safely on a branch and can't vanish. (If you had no new commits while detached — the usual case — skip this step.)
  3. Reattach to your real branch. Open the VCS widget → Local node → click main (or whichever branch you meant to be on) → Checkout.
  4. Confirm. The header now shows main, the ⚠ is gone. You're reattached. Done.
The 3-second rule Detached HEAD is harmless if caught immediately and a headache only if you keep working inside it. So build one reflex: after every checkout, read the header widget. Branch name = safe, carry on. ID + ⚠ = stop now and run the four escape steps above before doing anything else. Caught in the first three seconds, it's a non-event.
Why this kept happening to you Not carelessness — the interface invites it. The Log is the natural place to go "look back," and its Checkout option is right there in the right-click menu with no warning that it detaches. Nothing on screen told you what had happened or how to undo it. Now you have both: the cause (checking out a non-branch) and the escape (rescue if needed, then Local → branch → Checkout).

07 Five rules that prevent all of it

  1. Read the header before every action. Branch name = safe. ID + ⚠ = detached; fix first.
  2. Switch branches only via the VCS widget's Local node → click the name → Checkout. Never checkout a commit from the Log unless you mean to.
  3. Pull first, push last, every session, on whichever Mac you're at.
  4. Never commit workspace.xml or .idea/ files. Untick them; better, gitignore them.
  5. When confused, open the Log (⌥9). The commit graph is ground truth — where every label and HEAD really are, whatever a dialog claims.

08 Plain-language glossary

Repository (repo)
The project folder Git tracks, plus its full history.
Commit
A permanent saved snapshot of your files, with a message and an ID.
Branch
A movable label pointing at a commit. main is the default.
HEAD
Pointer to where you are now. On a branch normally; "detached" if pointing straight at a commit instead of a branch.
Checkout
Move HEAD somewhere — onto a branch (good) or onto a commit (detaches).
Origin
The nickname for your GitHub copy. origin/main = GitHub's version of main.
Fetch
Download GitHub's latest state without combining it. Always safe.
Pull
Fetch and then merge in one move. The merge half occasionally asks questions.
Push
Send your local commits up to GitHub.
Merge
Combine one branch's commits into another.
.gitignore
A text file listing files/folders Git should never track — used to exclude IDE clutter like workspace.xml.
Divergent
Two copies of a branch each have commits the other lacks; Git asks how to reconcile.
